## Freeze-window learnings (2026-04-22 → 2026-05-07)

A 15-day stabilization freeze ran inside the v7.5 closure window, originally scoped 30 days but lifted 15 days early after 5/5 hardening dims closed. The freeze did not produce features — it produced **operating discipline that v7.6 inherits**. Cross-ref: `feedback_two_week_freeze_synthesis.md`, `feedback_freeze_lifted_2026_05_07.md`.

### What shipped during freeze

Five stability fixes in 7 days, zero rollbacks:

1. **KB-visibility drift** — `1229 FS files orphan to DB + LIKE-substring search` → FTS5 + reindex ritual + `JARVIS_KB_MIRROR_DIR` env override. Commit `96933cb`.
2. **Scope safety nets** — algebra-progress + DENUE shell_exec gates. Commits `7aedc0f`, `a2767a6`.
3. **Hindsight provider migration** — qwen3p6 → minimax-m2p7 without retry storm.
4. **Write-tool friction** — jarvis*file*{write,update,delete,move} promoted to always-on; Rumi mitigation moved to tool-description + system-prompt. Commit `e70709c`.
5. **Fireworks .env propagation** — deprecated alias retry-storm caught + bounded.

### Three patterns observed

#### Pattern 1 — Narrow regex gates were structurally fragile

Three of the five fixes shared the same shape: a scope gate too tight to admit legitimate operator phrasing. Each fix widened the regex; the fifth capitulated and promoted four jarvis*file*\* tools to always-on.

**Lesson**: calling discipline belongs at the **tool-description and system-prompt layers, not at scope**. Scope decides what's _available_; the model decides what's _called_. The Rumi class of bug is a calling-discipline problem, not an exposure problem — fix it where calls are decided, not where tools are listed.

**v7.6 threshold**: if a fourth scope-gate-related incident hits, the architectural rethink jumps from P3 to P1. Until then it stays back-burner because per-incident fixes have been cheap. See `mc/docs/planning/scope-architecture-rethink.md`.

#### Pattern 2 — The Hindsight "promise gap"

"Jarvis remembers" was overpromised; what shipped is "Jarvis pattern-matches on similar-looking past text." Three concrete failure modes:

- Outcome-blind retrieval recycles failure narratives as recipes (Session 113-114 incident).
- Cross-encoder reranker is 99% of recall latency; 32% of recalls timeout at 5s budget.
- Bank growth without consolidation → reranker pool grows → either more latency or shrunk cap.

**Lesson**: don't pre-commit to a Hindsight architectural path. The strategic-options doc names the data we need; ship the instrumentation, accumulate ~2 weeks of measurement, then decide. Per-bank surgical demote (`HINDSIGHT_RECALL_DISABLED_BANKS=mc-jarvis`) is already a Path B opportunistic pilot. The global A/B/C decision is data-blocked through ~2026-05-21.

#### Pattern 3 — Silent state was the dominant failure class

Five concrete examples across the window:

- Fireworks deprecated alias still answered (caught only via dashboard $12 spike).
- 1229 KB orphan files invisible to Jarvis (caught only when operator asked "where is X").
- Hindsight dead worker leasing 6 jobs indefinitely (caught only via manual queue check).
- mc cost_ledger didn't see Hindsight spend at all (caught only via cross-reference of two dashboards).
- Cache-hit ratio drifted from 60% → 47.6% with no signal until manual audit (later corrected to actually 81.1% — _the audit itself was wrong_, but the absence of a passive metric was real).

**Lesson**: stability gains compound when **drift is observable**. Right now we only see what we've already had an incident about. **The cheapest two weeks of work are instrumentation moves, not architecture moves.**

### Operating principle inherited by v7.6

> **Instrument first. Decide later.**

The post-freeze queue (`next-sessions-queue.md`) is structured as: items #1-#6 are instrumentation/telemetry, #7-#10 are Hindsight-specific data collection, #11-#14 are deferred architectural items, #15 (Hindsight A/B/C) is gated on data from #4/#7/#8.

Closing v7.5 does NOT close this principle. v7.6 should adopt the same pattern: when tempted to jump to architectural decisions before instrumentation lands, the synthesis should pull us back. Two weeks of measurement is the load-bearing asset.

### Anti-patterns the freeze surfaced

- ⛔ **Don't add another regex arm to `scope.ts` without scoping the architecture** — three incidents in two weeks is structural, not local.
- ⛔ **Don't commit to a Hindsight path before #4/#7/#8 ship** — the strategic-options doc explicitly enumerates the prerequisites.
- ⛔ **Don't ship a fix without the corresponding signal** — every silent-state class above could have been caught earlier with a Prometheus metric or a `mc-ctl audit-claim`-style check. The fix without the signal just delays the next incident.

### What earned the early freeze lift

Per the operator's 2026-05-07 directive ("I think we can unfreeze Jarvis by now. I have enough evidence"):

- 5/5 hardening dims closed (`docs/audit/2026-04-22-*.md`).
- 5 distinct stability fixes shipped in 7 days without rollbacks.
- Re-benchmark still planned for 2026-05-22, independent of the lift.
- Re-arming template retained at `feedback_freeze_separation_policy.md` for future incident classes.

The "5/5 dims closed before lifting" pattern is documented as a precondition shape: don't lift a freeze with hardening work still red. v7.6 inherits this discipline.

## Recommended v7.6 spine

Natural anchors for the next phase. Note the ordering deliberately puts **instrumentation before architecture**, per the freeze-window operating principle:

1. **Hindsight A/B/C strategic decision** (data-ready 2026-05-21) — HARDEN vs DEMOTE vs REPLACE. See `next-sessions-queue.md` item #15 + `docs/planning/hindsight-strategic-options.md`. The two-week measurement window inherits the "instrument first, decide later" rule directly: don't pre-commit before #4/#7/#8 data accumulates.
2. **Drift-observability backfill** — every silent-state class the freeze surfaced (deprecated-alias retention, KB orphans, dead worker leases, missing cost telemetry, cache-hit ratio drift) deserves a passive Prometheus signal + alert. Pattern 3's "fix without signal just delays the next incident" applies: any v7.6 fix should ship with the metric that would have caught it earlier.
3. **`kb_entries.layer` schema add** — unlocks the v7.6-anchored cluster (two-layer memory, claude-mem labeling, ACE 3-agent loop).
4. **F7 wire-up of L4-L6 modules** — currently library-only. Plug `blackLittermanFromSignals → hierarchicalRiskParity` into `runAlphaCombination` + add `ReflectionRegistry` per specialist + insert `runAdversarialCritique` between PM draft and execution.
5. **Scope architecture rethink** (back-burner) — only if a fourth scope-gate-related incident hits. Pattern 1's threshold is explicit. Until then, calling discipline lives in tool descriptions + system prompts, not in regex.
6. **Tier-B unblocking** if operator provisions credentials.

These are not committed; they're the obvious next focal points. v7.6 scope to be defined separately. The discipline carryovers (audits, typecheck-first, full-suite-before-deploy, ship-it skill protocol) remain in force regardless of phase.

## Acknowledgement

The v7.5 release closed because it shipped what it set out to ship — financial stack, feature verticals, skill-evolution engine — and the leftovers cleanup today closed the actionable engineering subset. The residual catalog above is honest accounting, not aspiration: each item has either a real external dependency or an explicit telemetry trigger. No item is "we will get to this someday."

The 15-day freeze that ran inside this window is part of the closure too. It did not produce features; it produced operating discipline — three patterns named, three anti-patterns refused, and one principle ("instrument first, decide later") that v7.6 should re-read on its first day. The five stability fixes shipped during the freeze, the early lift, and the closure of the leftovers backlog are all the same shape of work: convert "we noticed an incident" into "we have a metric for the incident class" before declaring done.

What ships next is open. What ships well in v7.6 will be whatever inherits these learnings without being told to.
