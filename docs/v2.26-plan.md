# v2.26 Plan — From Q&A Assistant to Semi-Autonomous Agent

> Codename: `vigilant-forging-dijkstra`
> Goal: Close the reliability gaps, deepen COMMIT integration, and lay the foundation for Jarvis to act proactively — not just reactively.
> Sessions: 4–5 | Est. hours: 20–28h

## Thesis

89% of Jarvis's tasks are reactive chat. The infrastructure for autonomy exists
(rituals, schedules, proactive scans) but it's 7% of volume and has reliability
gaps (silent delivery failures, hallucinated execution, stale project context).
Before v3.0 production hardening, v2.26 closes these gaps and shifts the balance
toward autonomous value delivery.

COMMIT is the user's personal milestone system (Vision → Goal → Objective → Task).
It's the backbone of Jarvis's strategic awareness — not a feature to minimize but
the core context that makes autonomous actions meaningful. Half of COMMIT's 20
tools are unused not because they're unneeded, but because the system doesn't
create enough situations where they'd fire autonomously.

## Session 1: Delivery Guarantees + Token Budget (4–5h)

> No silent failures. No DashScope timeouts on heavy tasks.

### 1A. Required Tool Validation

Currently: scheduled tasks that need `gmail_send` complete silently if the LLM
never calls it (wrap-up produces text but no email). The delivery alert system
(v2.24) notifies after the fact but doesn't retry.

**Build:**
- `requiredTools` field on `TaskSubmission` — list of tools that MUST be called
  for the task to count as successful
- Post-execution check in dispatcher: if `requiredTools` specified and any are
  missing from `toolCalls`, mark task as `failed` with clear error
- Auto-retry once with an explicit system message: "You MUST call [tool] before
  completing this task. Your previous attempt did not call it."
- If retry also fails, alert via Telegram (existing alert system)

**Apply to:**
- Morning briefing: requires `commit__get_daily_snapshot`, `gmail_send`
- Nightly close: requires `commit__get_daily_snapshot`, `gmail_send`
- Scheduled email reports: requires `gmail_send`
- Image generation requests: requires `gemini_image`

### 1B. Token Budget Enforcement

Currently: prompt size grows unchecked in the main inference loop. At 25K+ tokens,
DashScope times out (caused 8 of 13 total failures).

**Build:**
- Hard cap on total conversation tokens before sending to inference (configurable,
  default 20K)
- When cap is hit mid-loop: aggressively truncate oldest tool results (not system
  prompt, not latest user message)
- Log a warning when approaching 80% of cap
- Different caps for fast (20K) vs heavy (40K) runners

### 1C. Provider Latency Tracking + Preemptive Failover

Currently: primary (GLM-5) is tried first, fails after 30s timeout, then falls
to qwen3.5-plus. Total wasted time: 30s per failure.

**Build:**
- Rolling latency average per provider (last 20 calls)
- If provider's average latency exceeds threshold (e.g. 15s), preemptively route
  to fallback without waiting for timeout
- Track and log provider health in `mc-ctl stats`
- Dashboard endpoint `/health` includes per-provider latency metrics

## Session 2: Project Entity + COMMIT Bridge (5–6h)

> Jarvis remembers everything about a project without being told twice.

### 2A. Projects Table

Currently: project data is scattered across `user_facts` (category: "projects")
as flat key-value pairs. No structure, no linking to COMMIT items, no changelog.

**Build:**
- `projects` SQLite table: `id, slug, name, description, status (active|paused|
  completed), urls (JSON), credentials (JSON, encrypted at rest), config (JSON),
  commit_goal_id (nullable FK to COMMIT goal), created_at, updated_at`
- `project_log` table: `id, project_id, action, details, created_at` — changelog
  of everything Jarvis does on a project
- Migration: auto-import existing `user_facts` with category "projects" into the
  new table (livingjoyfully, gemini key, etc.)

### 2B. Project Tools (3 new)

- `project_list` — List all projects with status, URLs, linked COMMIT goal
- `project_get` — Get full project details including credentials, config, recent
  log entries
- `project_update` — Update project config, credentials, status. Auto-logs changes

**ACI design:**
- Tool descriptions include: "When the user mentions a project by name, ALWAYS
  call project_get first to load its full context (credentials, URLs, config)"
- Auto-inject project context into system prompt when project name is detected
  in conversation (similar to `user_facts` injection)

### 2C. COMMIT ↔ Project Linking

- When a project is created, optionally link it to a COMMIT goal via
  `commit_goal_id`
- Project-aware daily snapshot: morning briefing includes project status alongside
  COMMIT hierarchy
- Nightly close: if a project has activity today (project_log entries), include
  a project status section

### 2D. Project-Aware Credential Resolution

- `wp_*` tools resolve credentials from projects table (not just `WP_SITES` env)
- `gemini_image` resolves API key from projects table
- Auto-detection safety net writes to projects table instead of user_facts
- Eliminates the "I don't have credentials" problem permanently

## Session 3: COMMIT Deep Integration (4–5h)

> COMMIT isn't just a data source — it's Jarvis's strategic compass.

### 3A. COMMIT-Aware Proactive Scanner

Currently: proactive scanner runs 4x/day and checks COMMIT snapshot. Output is
a simple nudge ("1 tarea vencida, 2 tareas hoy").

**Enhance:**
- **Stale goal detection**: goals with no objective/task updates in >14 days →
  nudge "Tu meta X no tiene actividad en 2 semanas. ¿La replanteamos?"
- **Objective completion tracking**: when all tasks under an objective are done,
  proactively suggest marking the objective as completed
- **Vision alignment check**: weekly (not daily) — are active goals still aligned
  with the stated vision? Flag orphan goals
- **Streak protection**: if the user's completion streak is at risk (no completions
  today by 6PM), send a gentle nudge with the easiest pending task

### 3B. COMMIT Auto-Actions (with confirmation)

Currently: Jarvis can update COMMIT items but only when explicitly asked. For
semi-autonomous operation:

- **Auto-complete recurring tasks**: when Jarvis observes evidence of completion
  (e.g., morning workout mentioned in chat, habit tracker data), proactively call
  `commit__complete_recurring` with confirmation
- **Auto-create tasks from conversations**: when the user says "tengo que..." or
  "necesito..." in chat, detect the implicit task and offer to create it in COMMIT
  under the right objective
- **Deadline proximity escalation**: tasks approaching due_date get auto-promoted
  to higher priority (with notification)

### 3C. COMMIT in Scheduled Tasks

Currently: scheduled tasks (oncology report, IA news) are standalone — they don't
connect back to COMMIT.

**Build:**
- Scheduled tasks can optionally link to a COMMIT objective
- On completion, auto-update the objective's status/notes
- On failure, create a follow-up task in COMMIT

### 3D. COMMIT Write Tool Enhancement

10 of 20 COMMIT tools have never been used. Not because they're unneeded but
because the LLM doesn't have enough context to use them autonomously. Fix:

- **create_goal**: add workflow guidance — "Before creating, call get_hierarchy
  to see existing goals. New goals must link to a vision."
- **create_vision**: protect against accidental creation — require confirmation,
  show existing visions first
- **bulk_reprioritize**: wire into nightly close — when tasks carry over, offer
  to reprioritize the next day's queue
- **complete_recurring**: wire into proactive scanner for auto-detection
- **create_journal_entry**: remains user-only (ritual respects this boundary)

## Session 4: Autonomous Loop Infrastructure (4–5h)

> The scaffolding for Jarvis to take initiative, not just respond.

### 4A. Initiative Engine

Currently: Jarvis is reactive (chat) or on a fixed schedule (rituals/cron). No
mechanism for context-triggered autonomous action.

**Build:**
- `initiatives` table: `id, trigger_type (schedule|event|threshold), trigger_config
  (JSON), action_template, project_id (nullable), active, last_fired_at, cooldown_ms`
- Trigger types:
  - `schedule`: cron expression (replaces current dynamic_schedules for new use cases)
  - `event`: fires when an event bus event matches criteria (e.g., "task.completed
    where task has tag 'livingjoyfully'")
  - `threshold`: fires when a metric crosses a boundary (e.g., "overdue_tasks > 3")
- Action templates: structured task descriptions that reference project context
- Cooldown: minimum time between firings to prevent spam

### 4B. Weekly Review Ritual

A new ritual type (not daily — weekly, Sunday evening):

- Full COMMIT hierarchy scan: orphan goals, stale objectives, overdue tasks
- Project health: each active project's status, last activity, blockers
- Week's accomplishments: completed tasks/objectives/recurring streaks
- Next week's focus: top 3 priorities derived from COMMIT + project state
- Email + Telegram delivery

### 4C. Conversation-to-Action Pipeline

When the user discusses something actionable in chat, Jarvis should detect it
and offer structured follow-up:

- **Pattern detection**: "necesito", "tengo que", "hay que", "debería",
  "recuérdame", "no olvides" → potential COMMIT task
- **Project detection**: mentions of project name → load project context
- **Follow-up offer**: "¿Quieres que cree una tarea en COMMIT para esto?" with
  pre-filled title, objective link, and due date suggestion
- Implementation: post-inference hook that scans the conversation for action
  patterns before sending the response

## Session 5: Tool Audit + Performance Polish (3–4h)

> Lean and fast, ready for v3.0.

### 5A. Tool Audit

102 tools registered, 49 used. Audit each unused tool:

| Action | Criteria |
|--------|----------|
| **Keep** | COMMIT tools (strategic backbone), Google tools (needed but rare) |
| **Consolidate** | `file_read`+`file_write`+`file_edit` could share a description prefix |
| **Lazy-load** | Tools that are only useful in specific contexts (coding, WP admin) should only load their full definition when scoped in |
| **Remove** | Truly dead tools with no foreseeable use case |

Target: 102 → ~85 registered, with lazy definitions saving ~4K tokens on average.

### 5B. Response Time Optimization

Average chat response: 25s. Target: <15s for simple queries.

- **Parallel enrichment**: memory recall + user facts + scope detection run in
  parallel (some may already be parallel — verify and optimize)
- **Fast-path for simple queries**: if classifier scores 0 and no tools are likely
  needed, skip tool definition injection entirely (text-only inference)
- **Streaming to Telegram**: emit first sentence as soon as it's generated, not
  after the full response completes

### 5C. Observability Prep (for v3.0)

- Container stderr → journalctl at INFO level (not DEBUG)
- Per-tool success/failure/latency tracking in SQLite
- `mc-ctl health` command: provider latency, tool success rates, scheduled task
  delivery rate, hallucination detection rate
