## Mission

v7.6 is a **reliability phase**, not a feature phase. The job is to take everything v7.0–v7.5 shipped and verify it actually works under production conditions — including the conditions where today's `prompt-enhancer-leakage` bug hid for weeks.

The orienting metric: **the next time we surface a class-of-bug like prompt-enhancer-leakage, instrumentation should catch it within 24h of the bad code merging — not weeks later by user complaint.**

---

## Anti-mission — what v7.6 will NOT do

These are deferred to v7.7+ regardless of how tempting they look mid-phase:

- **No new runners.** Five is enough.
- **No new tool categories.** 252 tools exist; coverage gaps in metadata are in scope, new tools are not.
- **No F7 financial-pipeline runner.** The L4–L6 modules (HRP / Black-Litterman / BM25 / adversarial-critic) are library-only and stay that way for v7.6. Composed library-level smoke tests are in scope; runner integration is a new capability.
- **No `kb_entries.layer` schema add.** Architectural addition; deferred even though `feedback_kb_visibility_drift.md` motivates it.
- **No substrate scouts adoption.** datagen / voyager / letta-sleeptime / smolagents / pheropath / langchain-ambient / devin-background / autogen-stall / mirix all stay reference-only.
- **No new skills.** Register the ones we already have; do not author new ones.
- **No prometheus loop architectural changes.** Plan-Execute-Reflect stays as-is. Instrumentation inside it is in scope; design changes are not.

If a candidate change feels like it might be a new feature, **it is**. Defer.

### Spine 1 — Gatekeeper audit pass — **CLOSED** (2026-05-08)

Apply the `prompt-enhancer-leakage` lens to every LLM-routing parser in the codebase. The pattern is: LLM returns one of N enum decisions → parser → router. Today's bug existed because one such parser had a `return raw` legacy fallback. Find the others.

**Audit candidates** (≥6, possibly ≥9):

| Module                                         | What to verify                                                        |
| ---------------------------------------------- | --------------------------------------------------------------------- |
| `messaging/scope-classifier.ts`                | Parse-failure path. Raw-output paths. Cold-start behavior.            |
| `dispatch/classifier.ts`                       | Decision shape on parse failure.                                      |
| `messaging/confirmations.ts`                   | Producer/consumer coupling — does the regex match what the LLM emits? |
| `messaging/fast-path.ts`                       | Tier-eligibility logic if LLM-driven.                                 |
| `runners/fast-runner.ts` (hallucination guard) | Pattern-match output paths.                                           |
| `messaging/post-filter.ts`                     | Output sanitization edge-case leaks.                                  |
| `prometheus/reflector.ts`                      | DimensionalCritique parsing — what if dim count varies?               |
| `prometheus/planner.ts`                        | Goal-graph parser parse-failure path.                                 |
| `messaging/normalize.ts`                       | Input normalization edge cases.                                       |

**Per-gatekeeper acceptance**:

- Typed enum return shape (no string returns that could carry raw LLM content)
- Parse-failure = hard-default (never `return raw` or echo unparsed content to user)
- Log counter that pin-tests against producer (RC4-style coupling test)
- Cold-start guard if applicable
- Risk carve-out if the gatekeeper has any "ergonomic relief" branch

**Estimate**: 6–9 audits × ~1h each. Likely +30–50 tests, +2–4 real bugs caught at the audit-bundle 33% hit-rate.

**Done when**: every gatekeeper has been audited and either has zero findings or has shipped the fix bundle. Tracked in `docs/audit/v7.6-gatekeepers.md`.

### Spine 2 — Observability backfill — **CLOSED** (2026-05-08)

Same lesson as today's bug: log counters can lie for weeks. Sweep every line that claims to count something and verify it counts what it claims.

**Search patterns to sweep**:

```
console.log.*\$\{.*\.length\}
console.log.*\$\{.*count\}
console.log.*Asking.*\$\{
console.log.*\bn=
```

For each match, verify either (a) it's already coupling-tested, or (b) add a producer/consumer test in the RC4 style.

**Tooling**: build `mc-ctl audit-log-counter` as a parallel to existing `mc-ctl audit-claim`. Takes a log-line pattern + producer-code reference, asserts they're consistent. Or simpler: a vitest pattern that grep-finds `console.log.*length` and requires a coupling test exists for each.

**Done when**: zero findings on a second-pass sweep. Audit log written to `docs/audit/v7.6-observability.md`.

## Operating rules (inherited from v7.5 + new from v7.6)

1. **Instrument first, decide later** (inherited; non-negotiable)
2. **Audit every bundle** — qa-auditor pass on every ship-it. The 33% real-bug rate justifies the cost. v7.5 ran 6 bundles, audit caught 2 real bugs (L4 NaN cascade, L6 token loss). Today's bundle ran audit, caught W1 risk=high carve-out as a real safety hole.
3. **Catalog-then-execute** — when a backlog spans multiple semantic domains, write the catalog as a separate commit before execution.
4. **Risk carve-outs everywhere** — every ergonomic-relief gate must have a high-risk override. Today's W1 was the canonical reminder.
5. **Producer/consumer coupling tests** — every observability surface needs both ends pinned in one test (today's W2 was the canonical reminder).
6. **Closure ritual every phase** — template exists; use it.
7. **No new features mid-phase** — if a candidate change feels like it might be one, it is. Add to v7.7 backlog and move on.

## What we're betting on

That **reliability is a phase, not a side-effect**. The 5 stability fixes during the freeze + the prompt-enhancer leakage today both demonstrate that what we already built has surface that hasn't been verified to work. v7.6 is the bet that 4 weeks of audit-and-pin will pay off more than 4 weeks of new building.

If at the end of v7.6 we've shipped zero new user-facing capability but have caught and fixed 8–12 real bugs across gatekeepers and observability, **that is a successful phase**. Keep that scoreboard in mind when the temptation to add a "small feature" hits in week 3.
