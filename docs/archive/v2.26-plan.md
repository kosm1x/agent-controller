# v2.26 Plan — Unification: One Brain, One Flow, One System

> Codename: `vigilant-forging-dijkstra`
> Goal: Merge COMMIT and Jarvis into a single autonomous system. COMMIT becomes the strategic UI. Jarvis becomes the intelligence layer. They share one brain, one event stream, one data flow.
> Repos: `kosm1x/agent-controller` (Jarvis) + `kosm1x/COMMIT-AI` (COMMIT)
> Sessions: 5–6 | Est. hours: 30–40h

## Thesis

Today, COMMIT and Jarvis are two separate systems connected by a brittle MCP
bridge. COMMIT has its own AI brain (Groq via Edge Function, 12 isolated
functions). Jarvis has its own brain (qwen3.5-plus/glm-5, 102 tools). They
operate on the same data but neither knows what the other said or did.

The result:
- Jarvis is 89% reactive (chat Q&A) because it has no triggers from where the
  user actually works (COMMIT UI)
- COMMIT's AI suggestions are shallow (one LLM call, no web access, no memory,
  no calendar, no project context)
- Credentials, project state, and user intent get lost between the two systems
- The user re-explains context that both systems already have

v2.26 ends this. After this release:
- COMMIT's AI functions route through Jarvis (one brain with full context)
- Data changes in COMMIT trigger Jarvis reactions (one event stream)
- Jarvis proposes changes that COMMIT surfaces (one flow)
- Projects, COMMIT goals, and Jarvis tasks are linked (one data model)
- COMMIT is the strategic UI where the user steers. Jarvis is the engine that
  moves.

## Session 1: Unified Data Layer (5–6h)

> Both systems know who changed what, and changes flow in real-time.

### 1A. Provenance — `modified_by` column [COMMIT repo]

Every change to COMMIT data should be attributable. Currently, Jarvis writes
via service role key and there's no way to distinguish user actions from agent
actions.

**Migration:**
```sql
ALTER TABLE visions ADD COLUMN modified_by TEXT DEFAULT 'user';
ALTER TABLE goals ADD COLUMN modified_by TEXT DEFAULT 'user';
ALTER TABLE objectives ADD COLUMN modified_by TEXT DEFAULT 'user';
ALTER TABLE tasks ADD COLUMN modified_by TEXT DEFAULT 'user';
ALTER TABLE journal_entries ADD COLUMN modified_by TEXT DEFAULT 'user';
```

Values: `'user'` (COMMIT UI), `'jarvis'` (agent), `'system'` (pg_cron, etc.)

**Apply to:**
- COMMIT frontend: always sets `modified_by: 'user'`
- MCP bridge write tools: always set `modified_by: 'jarvis'`
- Task pruning pg_cron: sets `modified_by: 'system'`

### 1B. Suggestions Table [COMMIT repo]

Jarvis proposes, user disposes. Instead of Jarvis directly modifying COMMIT
data for non-trivial changes, it writes to a suggestions queue. The COMMIT UI
surfaces these as actionable cards.

**Migration:**
```sql
CREATE TABLE agent_suggestions (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  type        TEXT NOT NULL, -- 'create_task', 'update_status', 'create_goal', 'reorder', 'archive'
  target_table TEXT,         -- 'tasks', 'goals', 'objectives', etc.
  target_id   UUID,          -- existing item to modify (null for creates)
  title       TEXT NOT NULL,  -- human-readable summary ("Crear tarea: Revisar GA4")
  suggestion  JSONB NOT NULL, -- full payload (what to create/update)
  reasoning   TEXT,           -- why Jarvis suggests this
  source      TEXT,           -- what triggered it ('proactive_scan', 'journal_analysis', 'conversation', 'weekly_review')
  status      TEXT DEFAULT 'pending', -- 'pending', 'accepted', 'rejected', 'expired'
  created_at  TIMESTAMPTZ DEFAULT now(),
  resolved_at TIMESTAMPTZ
);
```

**RLS:** User can only see their own suggestions.

**COMMIT UI changes:**
- Badge on nav showing pending suggestion count
- Suggestions panel (accept/reject/snooze)
- Accept → executes the suggested change + marks suggestion `accepted`
- Reject → marks `rejected` (Jarvis learns from rejections via Hindsight)

### 1C. Event Webhook [COMMIT repo]

A Supabase Edge Function that fires on database changes and notifies Jarvis.

**New Edge Function: `commit-events`**

Uses Supabase Database Webhooks (pg_net) or a polling approach:
- Trigger on INSERT/UPDATE to: `tasks`, `journal_entries`, `goals`, `objectives`
- POST to Jarvis's API: `http://localhost:8080/api/commit-events`
- Payload: `{ event, table, row_id, user_id, modified_by, changes }`
- Filters out `modified_by: 'jarvis'` to prevent echo loops

**Jarvis endpoint: `/api/commit-events`**

Receives COMMIT events and routes to the appropriate handler:
- `task.completed` → update project progress, check objective completion
- `journal.created` → trigger deep analysis (Session 2)
- `goal.status_changed` → update linked project, trigger celebration/next-steps
- `task.created` (by user) → check if it aligns with active goals
- `objective.completed` → suggest next objective or goal promotion

### 1D. API Key Auth for MCP Bridge [Jarvis repo]

Replace the service role key (bypasses all RLS) with a Supabase API key +
explicit user scoping. This is a security improvement and aligns with the
webhook auth pattern.

**Changes:**
- Generate a Supabase service account with limited permissions
- MCP bridge uses `SUPABASE_API_KEY` instead of `SUPABASE_SERVICE_ROLE_KEY`
- All queries continue to filter by `COMMIT_USER_ID`
- Write operations set `modified_by: 'jarvis'`

## Session 2: One Brain (6–7h)

> COMMIT's AI becomes Jarvis. Every AI interaction has full context.

### 2A. Jarvis AI Endpoint [Jarvis repo]

Expose Jarvis's intelligence as an API that COMMIT can call.

**New endpoint: `POST /api/ai`**
```json
{
  "function": "analyze_journal",
  "input": { "content": "...", "date": "..." },
  "context": { "user_id": "..." }
}
```

Supported functions (replacing COMMIT's 12):
| COMMIT function | Jarvis replacement | Added value |
|---|---|---|
| `analyzeJournalEntry` | Jarvis + memory + COMMIT goals | Connects emotions to goal progress, remembers past patterns |
| `extractObjectivesFromJournal` | Jarvis + COMMIT hierarchy | Places objectives under the right goal, avoids duplicates |
| `suggestObjectivesForGoal` | Jarvis + web + projects | Research-backed suggestions, aligned with project status |
| `suggestTasksForObjective` | Jarvis + calendar + projects | Calendar-aware due dates, project-aligned task design |
| `generateMindMap` | Jarvis + full hierarchy | Includes connections to existing goals/ideas |
| `completeIdea` | Jarvis + web search + memory | Expands with real-world research, not just LLM imagination |
| `findIdeaConnections` | Jarvis + COMMIT + projects | Cross-references ideas with active goals and projects |
| `generateDivergentPaths` | Jarvis + web + memory | Research-informed alternatives |
| `suggestNextSteps` | Jarvis + calendar + COMMIT | Time-aware steps with calendar integration |
| `generateCriticalAnalysis` | Jarvis + memory + projects | Includes past learnings and project constraints |
| `generateRelatedConcepts` | Jarvis + web search | Real articles/papers, not hallucinated resources |
| `transformIdeaText` | Jarvis (pass-through) | Same capability, unified context |

**Implementation:**
- New `src/api/ai-endpoint.ts` in Jarvis
- Dispatches to a specialized "commit-ai" tool set (fast runner, scoped tools)
- Injects full COMMIT context: daily snapshot + active goals + user facts + memory
- Returns structured JSON matching existing Zod schemas (drop-in compatible)

### 2B. COMMIT AI Service Rewire [COMMIT repo]

**Replace `callLLM()` in `aiService.ts`:**

```typescript
async function callLLM(prompt, temperature, max_tokens, ...): Promise<string | null> {
  // Try Jarvis first (full context)
  try {
    const response = await fetch(`${JARVIS_API_URL}/api/ai`, {
      method: 'POST',
      headers: { 'X-Api-Key': JARVIS_API_KEY },
      body: JSON.stringify({ function: currentFunction, input: { prompt }, context: { user_id } })
    });
    if (response.ok) return (await response.json()).content;
  } catch { /* fall through */ }

  // Fallback to Edge Function (offline/degraded mode)
  return callEdgeFunction(prompt, temperature, max_tokens);
}
```

**Key design decisions:**
- Graceful degradation: if Jarvis is offline, COMMIT falls back to the existing
  Edge Function (Groq). The user never sees a broken UI.
- COMMIT frontend remains a static SPA — no build-time dependency on Jarvis
- The Jarvis API key is stored as a Supabase Edge Function secret (not in frontend)
- A thin Edge Function `jarvis-proxy` handles auth + routing so the frontend
  never talks to Jarvis directly (CORS + security)

### 2C. Journal Analysis Deep Integration [Jarvis repo]

When the event webhook fires `journal.created`, Jarvis performs deep analysis
that COMMIT's isolated LLM never could:

1. Read the journal entry
2. Recall similar past entries from Hindsight memory
3. Cross-reference with active COMMIT goals + objectives
4. Check calendar for relevant events (meeting stress → journal anxiety?)
5. Check project status (LivingJoyfully deadline → journal overwhelm?)
6. Produce analysis: emotions, patterns, coping strategies, **goal alignment**
7. Write analysis back to COMMIT's `ai_analysis` table
8. If actionable insight detected → create suggestion ("You mentioned wanting to
   exercise more. Your goal 'Improve VO2 Max' has no tasks this week. Create one?")

This transforms journaling from "write → get emotion labels" to "write → get
strategic life insight with full context."

## Session 4: Strategic Autonomy (5–6h)

> Jarvis stops waiting to be asked. COMMIT becomes the steering wheel.

### 4A. Event Reactor [Jarvis repo]

Wire the webhook events (Session 1C) to intelligent reactions:

| Event | Reaction |
|---|---|
| `task.completed` | Check if all tasks under objective are done → suggest objective completion |
| `task.completed` (recurring) | Update streak, celebrate milestones (7, 30, 100 days) |
| `journal.created` | Deep analysis (Session 2C), detect action patterns |
| `goal.status_changed` to `completed` | Celebrate via Telegram, suggest archiving project, propose next goal |
| `objective.completed` | Suggest next objective, update project progress |
| `task.created` (by user) | Verify alignment with active goals (flag orphan tasks) |

### 4B. COMMIT-Aware Proactive Scanner

Enhance the 4x/day scanner with COMMIT-strategic intelligence:

- **Stale goal detection**: goals with no activity in 14+ days → suggestion
  "Tu meta X lleva 2 semanas sin movimiento. ¿La replanteamos?"
- **Objective completion tracking**: all tasks done → suggestion to mark complete
- **Vision alignment**: weekly check — are active goals still aligned with vision?
- **Streak protection**: if completion streak at risk (no completions by 6PM),
  nudge with the easiest pending task
- **Project-goal sync**: if a Jarvis project has activity but linked COMMIT goal
  doesn't, suggest updating the goal

### 4C. Conversation-to-COMMIT Pipeline

When the user discusses something actionable in Telegram chat:

- **Task detection**: "necesito", "tengo que", "hay que", "debería" → detect
  implicit task, match to best objective, create suggestion
- **Goal detection**: user discusses aspirational topic → check if a COMMIT goal
  exists, suggest creating one if not
- **Idea capture**: user mentions an idea or concept → suggest adding to COMMIT
  ideas library

### 4D. Weekly Review Ritual

New ritual (Sunday 8PM Mexico City):

1. Full COMMIT hierarchy scan: orphan goals, stale objectives, overdue tasks
2. Project health: each active project's status + linked COMMIT goal progress
3. Week's accomplishments: completed tasks, recurring streaks, journal consistency
4. Suggestions generated during the week: accepted vs rejected ratio
5. Next week's focus: top 3 priorities derived from COMMIT + project state
6. Email + Telegram delivery

### 4E. Required Tool Validation [Jarvis repo]

For scheduled/autonomous tasks, enforce that critical tools are called:

- `requiredTools` field on `TaskSubmission`
- Post-execution check: if required tool not in `toolCalls`, auto-retry once
- If retry also misses, alert via Telegram
- Apply to: morning briefing (`commit__get_daily_snapshot` + `gmail_send`),
  nightly close, scheduled reports

## Session 5: Reliability + Performance (4–5h)

> Fast, lean, ready for production.

### 5A. Token Budget Enforcement [Jarvis repo]

- Hard cap: 20K tokens (fast), 40K (heavy) before sending to inference
- Aggressive truncation of oldest tool results when approaching cap
- Different strategies: system prompt always kept, last user message kept,
  tool results get progressively shortened
- Prevents the DashScope timeout pattern (8 of 13 total failures)

### 5B. Provider Latency Tracking

- Rolling latency average per provider (last 20 calls)
- Preemptive failover when primary exceeds threshold (15s avg)
- Expose in `mc-ctl stats` and `/health` endpoint

### 5C. Tool Audit

102 tools registered, 49 used. Audit:

| Action | Tools | Rationale |
|--------|-------|-----------|
| **Keep** | All 20 COMMIT tools | Strategic backbone — v2.26 creates more trigger points |
| **Keep** | All 10 WP tools | Active project management |
| **Keep** | All Google tools | Email/calendar/drive integration |
| **Lazy-load** | 10 browser tools | Only load definitions when scope group activates |
| **Lazy-load** | 8 coding tools | Only load when coding scope activates |
| **Consolidate** | Evolution tools → single tool | 2→1, rarely used |

Target: ~85 active definitions per request (saving ~4K tokens avg).

**COMMIT tool ACI enhancement** (the 10 unused COMMIT tools):
- `create_goal`: add workflow — "call get_hierarchy first, link to vision"
- `create_vision`: require confirmation, show existing visions
- `bulk_reprioritize`: wire into nightly close for next-day queue rebalancing
- `complete_recurring`: wire into event reactor for auto-detection
- `create_journal_entry`: remains user-only (sacred boundary)
- `get_hierarchy`: use in weekly review and vision alignment checks
- `list_ideas`: wire into idea capture from conversations

### 5D. Response Time Optimization

- Parallel enrichment: memory recall + user facts + COMMIT snapshot in parallel
- Fast-path: if classifier scores 0 and no tools needed, skip tool definitions
- Streaming to Telegram: emit first sentence immediately
- Target: 25s → <15s average

### 5E. Observability

- Container stderr → journalctl at INFO (not DEBUG)
- Per-tool success/failure/latency tracking
- Suggestion acceptance rate tracking
- COMMIT event processing latency
- `mc-ctl health` with all metrics

## What This Changes for the User

**Before v2.26:**
- Open COMMIT app → manually manage goals, tasks, journal
- Open Telegram → ask Jarvis questions, get reports
- Context is split: Jarvis doesn't know you just completed a task, COMMIT
  doesn't know Jarvis just found a relevant article

**After v2.26:**
- Write a journal entry in COMMIT → Jarvis analyzes it with full life context
  (not just a generic LLM), connects your emotions to your active goals,
  suggests a specific task based on your calendar and projects
- Complete a task in COMMIT → Jarvis immediately notices, checks if the
  objective is now complete, updates your project status, prepares tomorrow
- Chat with Jarvis about an idea → Jarvis offers to add it to your COMMIT
  ideas library, linked to the right goal
- Morning briefing → includes project status alongside COMMIT hierarchy,
  with AI-generated strategic insight ("Your LivingJoyfully GA4 is live but
  you haven't published the '10,000 steps' post yet. It's blocking the
  'Content Pipeline' objective.")
- Jarvis proposes changes → you see them as cards in COMMIT, accept or reject
  with one tap. Jarvis learns from your rejections.

The user steers via COMMIT. Jarvis drives.
