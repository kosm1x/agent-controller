If a candidate item feels like a v8 capability, **it is**. Defer.

## The seven spines

Ordered by dependency. Spines 1-3 are independent and can fan out; Spines 4-6 layer on top; Spine 7 is the closure ritual. (Original Spine 7 — Hindsight A/B/C carry — collapsed; see v7.6-window deltas above.)

### Spine 1 — S2: Self-audit before reporting

**Spec**: `docs/planning/v8-substrate-s2-spec.md` (~5 days, 2 phases)

**Already shipped** (partial): `mc-ctl audit-claim` exists with utility/cache-hit/latency/cost stratification. The primitive is live. What's missing is the discipline-as-runtime: every report includes a `verified-against:` line citing a fresh data-source query.

**Phase 1**: harness + JSON schema enforcement on report shape. Borrows DATAGEN's note-agent JSON-schema enforcement.
**Phase 2**: retrofit `morning_brief` to emit the contract. Borrows Voyager's critic-as-write-gate.

**Activation gate** (per V8-VISION): zero "Audited?" cycles in a sprint of v7.7 reports — operator stops needing to ask.

**Retrofit targets** (post-v7.6 expansion): not just `morning_brief`. Add to the Phase 2 retrofit list:

- `morning_brief` — original target; aggregate-metric-citing daily report
- `community-manager` email reply path (`comunidades@mexiconecesario.org.mx` and any future org mailbox) — public-facing outputs MUST flow through critic-as-write-gate before send. Persona-grounded replies that cite numbers without `verified_against:` are exactly the failure class S2 prevents
- Closure docs themselves (V7.7-CLOSURE.md when it lands) — the v7.6 closure-audit pattern caught 4 fidelity issues; codifying `verified_against:` on closure-scoreboard rows makes the next closure audit faster

**Done when**: every fast/heavy/swarm runner output that quotes an aggregate metric carries `verified-against:`. Schema validation in CI. Community-manager email path enforces critic-as-write-gate. 6 open questions in spec, none blocking.

### Spine 2 — S3: Out-of-band drift detector

**Spec**: `docs/planning/v8-substrate-s3-spec.md` (~5 days)

3 small tables (`drift_signals`, `drift_alerts`, `baseline_history`) + cron evaluator + 12 seed signals across S1/S2/V8.2/V8.3/S5/infra, **plus a 13th seed signal added post-v7.6** for `mc_whatsapp_disconnects_total` (today's `aed876d` per pin `feedback_prometheus_counter_recovery_path` — "instrument all failure entry-points, not just `close`" became a load-bearing lesson before v7.7 even opened). **Activation gate**: any env edit bypassing the deploy script triggers a visible alert within 60s of next service interaction.

**Done when**: drift-signal table populated, evaluator running on cron (13 seed signals live), V8.1 morning-brief delivery hook in place. 8 open questions in spec, none blocking.

**Wins for v7.7 itself**: this spine catches drift while we're shipping S5 — the canonical case being a skill registry change that doesn't propagate. Ships drift-aware before the surface that needs it lands.

### Spine 3 — S5: Skills-as-stored-procedures

**Spec**: `docs/planning/v8-substrate-s5-spec.md` (~10 days, 5 phases)

Builds on the existing 57-skill shim at `src/db/skills.ts`. Composes DATAGEN's frontmatter contract + 3-level disclosure with Voyager's description-embedding + critic-as-gate + failed-tasks anti-list.

Skills become versioned procedures with frontmatter (`name`, `description`, `inputs`, `tests`), stored in `jarvis_files` under `skills/`, registered into the tool source manager at boot, runnable via `skill_run(name, args)`.

**Activation gate**: ≥5 production-grade skills with green test runs in the prior 7 days. (V8-VISION threshold; v7.7 gets us to that bar so V8.2 can compose against them.)

**Done when**: 5 phases shipped, 5 production skills registered (the existing HTML compose suite + 4 from current rituals — _registering, not authoring_), `skill_run()` invocable end-to-end. 7 open questions in spec, none blocking.

**Note**: this is the longest spine. Phase it explicitly so partial wins are commit-able. Phase 1 alone (registry + frontmatter contract) makes everything else tractable.

### Spine 4 — Conway Pattern 1: General-events middle layer

**Reference**: `reference_conway_2005_sms.md` — Conway 2005 Self-Memory System framework

**Why it's substrate**: V8.1 spec explicitly cites Pattern 1 as a structural prerequisite ("V8.1 cannot construct judgments without it"). Today, episodic memories live as raw events in the journal/cost_ledger; semantic memories live as facts in jarvis_files. There's no middle layer that aggregates events into "what's been happening this week" form.

**Shape**: derive a `general_events` view (or table) over journal + cost_ledger + recall_audit + decision_events. Time-windowed aggregations: per-day, per-week, per-project. Inputs to V8.1's morning-brief proactive scan.

**Done when**: V8.1's Phase 1 spec (Conway-pattern foundation, per V8.1 spec §"Phases 1-3 can ship independently") can run against this layer without further build-out.

---

### Spine 5 — Conway Pattern 2: Self-defining cohort

**Reference**: `reference_conway_2005_sms.md`

**Why it's substrate**: per V8-VISION §4, Pattern 2 is V8.2 prerequisite per Conway's empirical claim ("V8.1 reads as generic without it"). The system needs a stable model of _which projects, which contacts, which threads_ are the operator's identifiable cohort, so morning-brief judgments can speak in those terms instead of generic "you have 3 stalled tasks" language.

**Shape**: derive cohort membership from existing data — NorthStar projects, frequent CRM contacts, recurring conversation threads. Daily roll-up. No new schema if existing data supports it; one small denormalized table if not.

**Done when**: V8.1's Phase 2-3 (where cohort framing kicks in) can compose against the cohort definition without further design.

**Note**: this is the closest thing v7.7 has to "operator profile" — but kept narrow and substrate-shaped. Resist the urge to add personality / preference fields here. Those are V8.2 surface.

---

### Spine 6 — Conway Pattern 3: Named recall modes + correspondence audit

**Reference**: `reference_conway_2005_sms.md` + V8-VISION §9 trap warnings

**Why it's substrate**: V8-VISION §9 explicitly names "coherence drift toward confabulation" as a top trap. Pattern 3 is the codified defense: named recall modes (coherence vs correspondence) and a periodic correspondence-audit task. Without it, V8.2's evidence-cited proposals will drift confidently-wrong over months.

**Shape**: tag every recall result with mode (`coherence` = pattern-match historical, `correspondence` = check-current-state). Weekly correspondence-audit ritual that re-validates a sample of recently-cited memories against current reality. Surfaces stale memories before they end up in V8.2 proposals.

**Done when**: `recall_audit.mode` column live, weekly correspondence-audit cron emitting findings to `drift_signals` (composes with Spine 2). Stale-rate baseline measured.
