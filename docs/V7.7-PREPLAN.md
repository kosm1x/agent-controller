Initial framing from the 2026-05-08 reflection was **"memory done right"** — v7.7 as the bridge that fixes Jarvis's memory substrate before v8's relational arc lands.

On audit, that framing was based on incomplete knowledge. **Memory at the runtime level is largely already shipped:**

- ✅ S4 (cost_ledger v2) phases 1+2 shipped 2026-04-26 — universal inference path with cache breakdown end-to-end

**The actual remaining gap** between v7.6 closure and v8 capability work is the **S2/S3/S5 substrate items + Conway Patterns 1-3** — all specced 2026-04-30 at `docs/planning/v8-substrate-s{2,3,5}-spec.md` and `reference_conway_2005_sms.md`. v7.7 is therefore not "build new memory machinery" — it's **"ship the spec that's already on disk."** The pivot: we're not designing a bridge, we're laying it from drawings already authored.

## Anti-mission — what v7.7 will NOT do

- **No V8.1 / V8.2 / V8.3 capability work.** v7.7 lays the foundation; the capability landings are v8.0+. If a substrate item creeps into delivering capability behavior, defer the capability and ship just the substrate primitive.
- **No new tools.** 252 is enough. S5 _registers_ skills as first-class capabilities; it does not author new ones.
- **No new runners.** Five is enough.
- **No memory architecture changes.** pgvector + Ebbinghaus + unified FS + kb_entries layer hierarchy already work. Don't redesign them.
- **No substrate scout adoption mid-phase.** Same defer policy as v7.6. Reference docs stay reference.
- **Operator-profile: skeleton only — deliberate carve-out from the original anti-mission.** Resolved 2026-05-18 (Q3): land a thin scaffold in v7.7 that V8.2 can extend, rather than dropping the surface entirely. Constraint: scaffold MUST define only the storage shape + read API; NO inference / NO personality / NO preference fields / NO write-side population beyond what Conway Pattern 2 cohort roll-up already produces. V8.2 owns the semantic layer on top. Test for compliance: can V8.2 redefine the schema's value-side without migrating data? If yes, the scaffold is thin enough.
- **No closure-ritual automation as a feature.** v7.5 + v7.6 closures will have run by then; the template is proven manual. Automate it in v7.8 if it's still painful.

If a candidate item feels like a v8 capability, **it is**. Defer.

**Spec**: `docs/planning/v8-substrate-s2-spec.md` (~5 days, 2 phases)

**Already shipped** (partial): `mc-ctl audit-claim` exists with utility/cache-hit/latency/cost stratification. The primitive is live. What's missing is the discipline-as-runtime: every report includes a `verified-against:` line citing a fresh data-source query.

- Closure docs themselves (V7.7-CLOSURE.md when it lands) — the v7.6 closure-audit pattern caught 4 fidelity issues; codifying `verified_against:` on closure-scoreboard rows makes the next closure audit faster

**Done when**: every fast/heavy/swarm runner output that quotes an aggregate metric carries `verified-against:`. Schema validation in CI. Community-manager email path enforces critic-as-write-gate. 6 open questions in spec, none blocking.

### Spine 4 — Conway Pattern 1: General-events middle layer

**Reference**: `reference_conway_2005_sms.md` — Conway 2005 Self-Memory System framework

**Why it's substrate**: V8.1 spec explicitly cites Pattern 1 as a structural prerequisite ("V8.1 cannot construct judgments without it"). Today, episodic memories live as raw events in the journal/cost_ledger; semantic memories live as facts in jarvis_files. There's no middle layer that aggregates events into "what's been happening this week" form.

**Shape**: derive a `general_events` view (or table) over journal + cost_ledger + recall_audit + decision_events. Time-windowed aggregations: per-day, per-week, per-project. Inputs to V8.1's morning-brief proactive scan.

**Done when**: V8.1's Phase 1 spec (Conway-pattern foundation, per V8.1 spec §"Phases 1-3 can ship independently") can run against this layer without further build-out.

### Spine 5 — Conway Pattern 2: Self-defining cohort

**Reference**: `reference_conway_2005_sms.md`

**Why it's substrate**: per V8-VISION §4, Pattern 2 is V8.2 prerequisite per Conway's empirical claim ("V8.1 reads as generic without it"). The system needs a stable model of _which projects, which contacts, which threads_ are the operator's identifiable cohort, so morning-brief judgments can speak in those terms instead of generic "you have 3 stalled tasks" language.

**Shape**: derive cohort membership from existing data — NorthStar projects, frequent CRM contacts, recurring conversation threads. Daily roll-up. No new schema if existing data supports it; one small denormalized table if not.

**Done when**: V8.1's Phase 2-3 (where cohort framing kicks in) can compose against the cohort definition without further design. **Plus** (per Q3 resolution 2026-05-18): operator-profile skeleton table shipped alongside the cohort roll-up. Schema only — `operator_profile` table with `(cohort_member_id, attribute_key, attribute_value, written_at, written_by)` — write-side populated ONLY by Conway Pattern 2 cohort roll-up (no inference, no LLM-derived fields). V8.2 owns what `attribute_key`/`attribute_value` mean semantically. v7.7's contract is just "the bone structure exists and V8.2 can extend without migration."

**Anti-mission test for the skeleton** (per §44 amendment): can V8.2 redefine the value-side semantics without migrating data? If yes, the skeleton is thin enough. If no, we shipped too much — revert.

---

### Spine 6 — Conway Pattern 3: Named recall modes + correspondence audit

**Reference**: `reference_conway_2005_sms.md` + V8-VISION §9 trap warnings

**Why it's substrate**: V8-VISION §9 explicitly names "coherence drift toward confabulation" as a top trap. Pattern 3 is the codified defense: named recall modes (coherence vs correspondence) and a periodic correspondence-audit task. Without it, V8.2's evidence-cited proposals will drift confidently-wrong over months.

**Shape**: tag every recall result with mode (`coherence` = pattern-match historical, `correspondence` = check-current-state). Weekly correspondence-audit ritual that re-validates a sample of recently-cited memories against current reality. Surfaces stale memories before they end up in V8.2 proposals.

**Done when**: `recall_audit.mode` column live, weekly correspondence-audit cron emitting findings to `drift_signals` (composes with Spine 2). Stale-rate baseline measured.

### Part B — continuity tooling (Q4 absorption)

Codify the three-time-proven manual ritual into commands:

- **`mc close-phase <name>`** — automates steps 1-3: writes a draft `<NAME>-CLOSURE.md` skeleton from the spine table in `<NAME>-GUIDE.md`, drafts the watchlist reframe at `docs/planning/<name>-leftovers-queue.md`, creates the annotated git tag draft (does NOT push — operator review gate).
- **`mc resume <phase>`** — re-loads the phase's open queue + watchlist + last commit ledger into the session brief, so v7.8 (or whatever comes after a break) opens with context already paged in.
- **`mc audit-closure <closure-doc>`** — wraps the qa-auditor invocation pattern used in v7.5 and v7.6 closures (cross-check spine commits, sum scoreboard math, verify anti-mission held, validate memory-pin existence). The pattern recurred identically across both closures — the automation surface is well-defined.

Constraint: these tools MUST NOT execute git operations (commit, tag, push) without operator confirmation. The ritual exists to make closures correct, not to make them automatic. Failure mode to avoid: a typo'd `mc close-phase` that auto-pushes a half-baked closure tag.

**Done when**: v7.7 closure itself uses the new commands at least once end-to-end. The drafted CLOSURE.md still gets manual editing before tag — that's the design.
