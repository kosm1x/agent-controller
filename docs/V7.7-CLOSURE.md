## Patterns observed in v7.7

### Pattern 1 — Spec-vs-codebase reconciliation

Capability specs (V8.1 §5, the Conway reference) repeatedly named tables,
columns, and types that did not match codebase reality — `vec0` virtual
tables vs the BLOB-embedding convention, `FLOAT[1024]` vs 1536-dim Gemini,
`journal`/`decision_events` tables that do not exist, the un-ALTERable
`jarvis_files.qualifier` enum. Every spine designed to ground truth and
logged the deviation in its impl-delta. A spec is design intent, not an API
contract — verify against the live schema before building.

### Pattern 2 — Verify live; the first real run is an audit gate

Three spines shipped a bug invisible to unit tests and the auditor that
only the first live run exposed: Spine 4's `.env`-sourcing abort, Spine 5's
objective filter wrongly admitting CHANGELOG files, Spine 6's `recall_audit`
dormancy. Closure-stage live verification (`deploy` + run the thing + read
the output) caught all three. Unit tests use synthetic inputs; the auditor
has no live data; the live run has both.

### Pattern 3 — Compose onto existing substrate; disabled-pending is honest

Spine 6's correspondence audit is a `weekly`-cadence drift signal on Spine
2's substrate — not a new cron. When a signal's data source is not yet
live, it ships `enabled=0` with a documented activation trigger (Spine 2's
10/13, Spine 6's 1) — never `enabled=1`-pretending-to-work. Substrate is
allowed to ship ahead of its consumer; it is not allowed to pretend.
