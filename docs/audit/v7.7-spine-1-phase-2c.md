### Rejected finding (R1-C1)

> **R1-C1 (qa-auditor)**: "CLI exits 0 on V7.6-CLOSURE.md despite 10 unverified claims — the convention's tag-gate is broken on the proof case."

**Verdict: NOT REPRODUCED.** Re-ran the exact command from the auditor's repro line — `EXIT=1` as expected. Logs above confirm. The auditor's reported `EXIT=0` was uncorroborated; possibly a tsx-runtime conflation or transient state. The CLI's exit code path is correct:

```ts
const report = validateClosureDoc(argvPath, contents);
process.stdout.write(renderReport(report));
process.exit(report.exitCode);
```

`process.exit` is synchronous and reliably propagates the exit code; `process.stdout.write` to a TTY/pipe is line-buffered, not async-deferred. No race.

**Lesson** (codified separately): qa-auditor's claim of having empirically verified a finding does not substitute for re-running it. The R1 trust model is "auditor surfaces hypothesis; ship rejects with proof." Rejecting on proof is fine; folding without verifying would have been the failure mode.

## Tier-1 → R1-only downgrade rationale

R2 catches "bugs INSIDE R1 fixes." The two non-trivial fixes (W1 regex grouping, W2 heading-depth condition) are mechanical and locked by regression tests. The "rejected C1" path actually adds confidence — the bundle's correctness on the proof case is empirically demonstrated by `EXIT=1` over the V7.6 doc.

Where R2 WOULD have been mandatory: any change to the claim-detection grammar that could leak adjacent surfaces. The current bundle confines markdown parsing to one heuristic library, well-tested.
