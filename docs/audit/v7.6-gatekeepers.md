## #3 — `messaging/confirmations.ts`

**Audited**: 2026-05-08
**Verdict**: ❌ Real finding — producer/consumer drift across two parallel regex sets. Canonical RC4-style mismatch.

| Lens                                  | Result | Evidence                                                                                                                                                                                                                                                              |
| ------------------------------------- | ------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1. Typed enum return                  | ✅     | `"confirm" \| "decline" \| null` (line 110-112). Strict union, no string passthrough.                                                                                                                                                                                 |
| 2. Parse-failure hard-default         | ✅     | No-match → `null`. Caller treats null as "not a confirmation" and proceeds to normal routing.                                                                                                                                                                         |
| 3. Log counter coupling               | ❌     | No counter. **More importantly: a SECOND CONFIRM_PATTERN exists at `runners/fast-runner.ts:104`** for the deletion-confirmation flow, and the two diverge. See F5 below.                                                                                              |
| 4. Cold-start guard                   | N/A    | Stateless regex.                                                                                                                                                                                                                                                      |
| 5. Risk carve-out on ergonomic relief | ⚠️     | 60-char threshold (line 115) is uniform. A high-stakes destructive operation (delete N tasks, send mass email) gets the same lax matcher as a benign one. No carve-out for `requiresConfirmation && destructiveHint` tools to require stricter / longer-form replies. |

**Findings**:

- **F5 (HIGH — real bug class)** — Two parallel CONFIRM_PATTERNs exist in the codebase, serving overlapping but divergent purposes:

  | Location                                     | Pattern coverage                                                                                                                                                         |
  | -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
  | `messaging/confirmations.ts:99`              | `s[ií]\|dale\|hazlo\|procede\|adelante\|ok\|okey\|va\|confirmo\|env[ií]a(lo)?\|m[aá]ndalo\|ejecuta\|claro\|por\s*favor\|yes\|go\|confirm\|send\s*it\|do\s*it\|approved?` |
  | `runners/fast-runner.ts:104` (deletion-only) | `s[ií]\|confirmo\|dale\|ok\|yes\|hazlo\|adelante\|procede\|proceed\|confirm\|bórrala\|bórralas\|elimínalas?\|go ahead\|do it\|haz(?:lo)?\|claro`                         |

  Drift directions:
  - `confirmations.ts` has `m[aá]ndalo, env[ií]a(lo)?, ejecuta, send_it, approved` (production verbs).
  - `fast-runner.ts` has `bórrala, bórralas, elimínalas, proceed, go ahead, haz(lo)?` (deletion-clitic forms).
  - **Neither has** `súbelo/súbela, créalo/créala, lánzalo/lánzala, tráelo/tráela, ágregálo/agrégala, cámbialo/cámbiala` — common Spanish imperative-clitic confirmations the user naturally produces in response to LLM prompts like "¿Subo el archivo?" / "¿Creo el evento?" / "¿Lanzo el experimento?".
  - **No coupling test** asserts that the LLM-side prompts (system prompt + tool descriptions that elicit confirmation questions) align with what either regex set will accept.

  This is the canonical RC4 mismatch: producer (LLM-asked confirmation) and consumer (regex matching user reply) drift silently. When they diverge, the user sees Jarvis asking "¿Lo subo?", they reply "Súbelo", and the confirmation matcher returns `null` → the pending operation never executes → user retypes longer phrasing → friction tax.

  **Adjacent context**: This is the bug class implied by friction-pickup #1's "short Spanish imperatives" pattern, but `procede`/`dale`/`hazlo` already match in BOTH regexes. So friction-pickup #1's specific incident is NOT this F5 — it's a different layer (LLM didn't gate on CONFIRMATION_REQUIRED, or the user message wasn't routed to the confirmation matcher because no pending was stashed). F5 stands as its own real bug.

- **F6 (MEDIUM)** — Uniform 60-char threshold. Destructive ops (delete, send, push) should require stricter matching — e.g. exact-anchor or longer-form reply — to prevent typo'd "noo k" or "no estoy seguro pero ok" from accidentally confirming a high-risk operation. Carve-out missing.

- **F7 (LOW)** — No log counter for which pattern matched / which path was rejected. Spine 2.

**Fix bundle scope** (proposed, not shipped this round — operator should weigh in on regex changes that touch destructive-op confirmation flow):

1. Unify the two regexes behind a shared `CONFIRMATION_VERBS` constant + small variants per call site.
2. Add the missing imperative-clitic forms (súbelo, créalo, lánzalo, tráelo, etc.).
3. Add a coupling test in `confirmations.test.ts` that:
   - Pins the LLM-side prompt forms (a representative sample harvested from journalctl + tool descriptions that elicit confirmation).
   - Asserts every natural reply to those forms matches CONFIRM_PATTERNS.
   - Fails the test when a new tool with `requiresConfirmation: true` adds a producer-side phrasing not covered by the consumer.
4. Add risk carve-out: if the pending op's tool has `destructiveHint: true`, require a stricter match (full-word anchor, no substring; minimum reply length 2 chars; no character-class slop).

**Recommend**: surface F5 + F6 to operator before regex change. Confirmation surface is touchy; a wider regex risks accidentally confirming on incidental reply text.

### Round 2 audit findings + fixes shipped 2026-05-08

qa-auditor returned **FAIL** with 2 critical issues. Both legitimate, both addressed:

**C1 — `va` / `go` over-fire in strict mode** (`confirmation-verbs.ts:19` pre-fix)

- Pre-fix: `GENERIC_CONFIRM_SRC` included bare `va` and `go`. Strict mode draws only from GENERIC, so `"va para allá"`, `"go away"`, `"go home"` all confirmed when a destructive op was pending — exact bug class the audit was built to find.
- Fix: removed `va` and `go` from GENERIC. `vale` moved to ACTION (lax-only) since it can appear mid-sentence as filler. `go ahead` is in EN_CONFIRM_SRC; bare `go` is no longer accepted.
- Tests: 5 new C1 regression-guard cases (`"va para allá"`, `"va a casa"`, `"go away"`, `"go to hell"`, `"go home"`) all assert `null` in strict mode.

**C2 — Strict mode rejected legitimate destructive confirmations** (`confirmation-verbs.ts:93-104` pre-fix)

- Pre-fix: strict mode used `[GENERIC_CONFIRM_SRC]` only. `"Bórralo"` (delete-clitic confirming a delete op) was rejected → user sees no acknowledgment, retries, gives up. UX regression on the most common destructive flow.
- Fix: introduced `DESTRUCTIVE_CLITIC_CONFIRM_SRC` (verbs semantically aligned with destructive ops: `b[oó]rr|elim[ií]n|qu[ií]t|remu[eé]v|desact[ií]v|deshabil[ií]t`). Strict mode now accepts GENERIC + this subset, so `"Bórralo"` confirms but `"Súbelo"` (verb/op-type mismatch) still doesn't.
- Tests: 13 new positive cases for destructive clitics in strict mode + 5 new negative cases for non-destructive clitics in strict mode.

**W3 — `procédelo` not covered**: added `proced(?:elo|ela|elos|elas)` to ACTION_CONFIRM_SRC (lax-only, since strict already covers the destructive semantics it would otherwise satisfy).

**W1 — strict-mode false-positive coverage**: closed by C1 regression-guard tests above.

**W2 — Router integration test for `destructiveHint` lookup chain**: deferred. The chain `pendingConf.toolName → toolRegistry.get → ?.destructiveHint` is short and the fallback (no tool found → no `destructiveHint` → lax mode) is safe. Worth adding eventually but not blocking this bundle.

**Final verification**: typecheck clean. **4470 / 4470 tests pass** (+66 new vs pre-Spine-1 baseline). qa-auditor 2-round cadence captured what the 1-round bundle missed — exactly the pattern documented in `feedback_strict_mirror_sync_pattern.md` (round 1 catches concrete bugs; round 2 catches contract-level issues).

### Round 2 follow-up audit (post-commit `90ebc31`) — additional findings shipped 2026-05-08

A second qa-auditor pass on the committed bundle found 2 more critical issues + 1 standards violation. All addressed:

**C3 (load-bearing) — `hasUserConfirmedDeletion` regressed to lax** (`runners/fast-runner.ts:107`)

- Bundle's stated purpose: eliminate drift between the two confirmation regexes. `fast-runner.ts:104`'s pre-bundle hand-written regex was narrow and destructive-aligned (`bórrala/elimínalas`). Post-bundle it pointed at `buildConfirmRegex("lax")` — accepting `súbelo`, `créalo`, `lánzalo` etc. for a deletion two-step. This is the exact verb/op-mismatch the strict-mode carve-out was added to prevent. Drift moved, not closed.
- Fix: changed to `buildConfirmRegex("strict")`. Deletion two-step is inherently destructive; strict mode is the right vocabulary.

**C3 cascading refinement** — round-1 strict mode was over-aggressive

- Investigating C3 surfaced an existing test `runners/fast-runner.test.ts:723` that expects `Dale` to confirm a deletion two-step. Round-1's strict mode dropped ALL action verbs, which would have broken this. Reframe: action verbs (`dale`, `hazlo`, `procede`, `adelante`) are op-INDIFFERENT — they mean "go ahead with whatever you proposed" regardless of op type. The verb/op-mismatch class only applies to verb-typed clitics (`súbelo` to delete op).
- Fix: refined `buildConfirmRegex("strict")` to be `GENERIC + ACTION + DESTRUCTIVE_CLITIC` (was `GENERIC + DESTRUCTIVE_CLITIC` post-round-1). Existing fast-runner test passes; non-destructive clitics still gated.
- `vale` was considered for ACTION but EXCLUDED — same false-positive risk as `va`/`go` removed in C1 ("vale la pena" / "no vale" read as filler).

**C4 — `para` decline over-fires on Spanish preposition** (`confirmation-verbs.ts:128` pre-fix)

- Pre-fix: `DECLINE_SRC` included bare `para`. In Spanish overwhelmingly the preposition "for/towards" — `"para mí está bien"` (24 chars) registered as decline.
- Fix: replaced bare `para` with compound `para\s+ya`. Imperative "stop" still covered by `detente`, `alto`, `stop`. Added `alto` (Spanish "halt").
- Tests: 4 new C4 regression-guard cases (`"para mí"`, `"para allá"`, `"para que veas"`, `"para él"`) all assert `null`.

**S1 — Audit doc inflated hit-rate framing** (this section, pre-update)

- Pre-update: scoreboard claimed "3 real bugs in 3 audits = 100%". But C1 and C2 were SELF-INFLICTED regressions introduced by the round-1 bundle, not pre-existing bugs. F5 was the only pre-existing bug.
- Honest scoreboard:

  - C1 / C2 (2) — bundle-regressions caught in round-2 before commit

- True audit value: 1 pre-existing bug + 5 bundle-regression catches across 2 audit rounds. The 2-round cadence is doing the work; the bundle iteration is the cost of a cleanly-shipped change.

**W1 — "producer/consumer coupling" test was misnamed** (`confirmations.test.ts:265`)

- Pre-fix: the test pinned hardcoded LLM-prompt → reply pairs but only ever asserted `detectConfirmationResponse(reply)`. The `llmPrompt` field was never passed through any producer code. It was a duplicate of the lax positive list with prose context.
- Fix: renamed `describe("producer/consumer coupling")` → `describe("reply vocabulary by op family")` with comment explaining what would be needed for a real coupling test (extract LLM-prompt strings from tool descriptions; assert each elicits a matching reply via mocked LLM or grep-extracted fixtures).

**W2 — Tool-annotation snapshot drift**: deferred with explicit note. `pendingConfirmations.ts` stores `{toolName, args, timestamp, summary}` only; `router.ts:1418` re-reads `destructiveHint` at confirm time. With v7.6 Spine 4 (MCP annotation push) coming, snapshot the destructive-hint at stash time instead. Tracked as a v7.6 leftover.

**R2 — Syntactic regex-source guards**: 3 new tests (`confirmations.test.ts:323-365`) trip CI if a future diff:

- Adds non-destructive clitic stems back into strict mode
- Removes destructive clitic stems from strict mode
- Resurrects bare `para` in DECLINE_SRC
