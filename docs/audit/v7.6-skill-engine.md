## Architecture decisions

**Static imports vs filesystem scan**. The brief said "filesystem-scan discovery" but the implementation uses TS imports of upstream `as const` arrays. Reasoning: static imports are simpler, tsc-verified, defensible against rename, and meet the spirit of the criterion ("registering existing scaffolding"). FS scanning would have added node:fs read paths at boot for zero functional gain.

**`requiresTool` gating**. Round-1 audit H1 caught that the section was being pushed unconditionally to P3, advertising tools the LLM didn't have in scope — direct violation of `prompt-sections.ts:65-66` "no menciones herramientas que no están en tu lista." Fix: each `FirstPartySkill` declares its `requiresTool`; `listSkillsForTools(tools)` filters by Set membership; `availableSkillsSection` returns "" when filtered list is empty so the P3 push auto-omits.

**Bind to canonical tool names**. Round-2 audit M4 flagged the magic strings `"video_html_compose"` and `"diagram_generate"` as silent rename hazards. Initial fix imported the tools directly (`videoHtmlComposeTool.name`); however that polluted the catalog's import chain with `child_process` / `playwright` deps, breaking partial-mock tests in `src/rituals/safeguard-tags.test.ts`. Final fix: catalog keeps the strings as named module-private constants (`VIDEO_HTML_COMPOSE_TOOL_NAME`, `DIAGRAM_GENERATE_TOOL_NAME`) and `catalog.test.ts` asserts equivalence to the canonical `Tool.name` fields via a test-time dynamic import. A rename of either Tool's `name` still produces a loud test failure — at the cost of one test boundary instead of typecheck-time enforcement. Trade-off documented inline in `catalog.ts:64-72`.

**DB-saved skills excluded from system prompt**. The 200+ DB-saved skills are surfaced via `mc-ctl skills` and the existing `skill_list` tool, but NOT mixed into `availableSkillsSection`. Reason documented in `catalog.ts:21-24`: keeping first-party scaffolding distinct from runtime-mutated skills prevents one layer from masking the other.

## Lessons

1. **Asymmetric upstream coupling is invisible until you write the catalog**. html-compose had its `{ surface, label, content }[]` array from day one (v7.4.3.6 L1); svg-diagram exposed only the raw content strings. The catalog's job is to re-expose, not author. Round-1 M3 forced the symmetry retroactively.

2. **Engine + payload separation is load-bearing**. The skill engine gate (v7.4.3.6 L1) shipped 6 weeks before the payload could be wired in. The reason wasn't "no skills" — it was "no surface to point the engine at." Spine 5's contribution is the **surface**, not the skills.

3. **Tool-name binding via import beats magic strings**. Round-2 M4 took 2 lines (one import, one `tool.name` read per skill) and converted a silent-fail rename hazard into a typecheck failure. Apply this pattern wherever a downstream layer references a tool by string.

4. **Round-2 catches deferred recommendations from round-1**. Round-1 made three recommendations (single-source-of-truth tool name, e2e placement test, audit doc); round-2 caught two of those as MISSED rather than letting them silently slip into "we'll get to it." Closure ritual depends on round-2 reading round-1's recommendations as test cases.

5. **`Object.freeze` requires both array AND entries**. Spreading frozen entries into a non-frozen array breaks the contract — `array.pop()` succeeds even when entries are frozen. Symmetric freeze needed.
