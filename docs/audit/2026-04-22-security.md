## Round 1 — Criticals (6)

### C-INJ-1 — claude-sdk tool results not sanitized

**Evidence**: `src/inference/claude-sdk.ts:85-112` `wrapTool()` returns `{ content: [{ type: "text", text: result }] }` directly from `toolRegistry.execute()`. `src/inference/adapter.ts:1679` (OpenAI path) calls `sanitizeToolResult(toolCall.function.name, result)`. SDK path had no equivalent.

**Impact**: With `INFERENCE_PRIMARY_PROVIDER=claude-sdk` as primary since 2026-04-22, all prompt-injection via tool output (web*read, exa_search, seo*\*) reached Sonnet unfiltered. Role markers (`SYSTEM:`, `[INST]`, `<system>`) not defanged.

**Fix**: `src/inference/claude-sdk.ts` — imported `sanitizeToolResult`, wraps `result` before sending to SDK.

### C-PATH-1 — file_read unguarded

**Evidence**: `src/tools/builtin/file.ts:149-172` `fileReadTool.execute()` calls `readFileSync(path, "utf-8")` with zero validation. `file_write` has `isWriteAllowed()`; `file_read` had nothing.

**Impact**: LLM (or prompt-injected payload via tool output → conversation history) could call `file_read` with `/root/.claude/.credentials.json`, `/etc/shadow`, `/opt/supabase/docker/.env`. Direct secret-exfil.

**Fix**: wired `validatePathSafety(path, "read")` into `fileReadTool`.

### C-PATH-2 — validatePathSafety read mode skipped dangerous files

**Evidence**: `src/tools/builtin/immutable-core.ts:166` `if (operation !== "read" && isDangerousFile)` — dangerous-file check was skipped for read mode. Same pattern at line 173 for dangerous directories.

**Impact**: Even tools that DID call `validatePathSafety(..., "read")` (file-convert, code-editing, market-chart-patterns) were getting `safe: true` on credential-bearing paths.

**Fix**: added `READ_BLOCKED_PATHS` + `READ_BLOCKED_BASENAMES` lists in immutable-core.ts; `validatePathSafety` now enforces them regardless of operation. Round 2 later extended with `realpathSync` symlink check.

### C-PATH-3 — wp_media_upload reads arbitrary local file

**Evidence**: `src/tools/builtin/wordpress.ts:791-818` pre-fix: `imageUrl.startsWith("/")` set `isLocalFile = true`; then `readFileSync(filePath)` on the absolute path; the bytes were uploaded to WordPress via the media endpoint.

**Impact**: LLM setting `image_url="/root/.claude/.credentials.json"` would upload the credential file to the external WordPress site (publicly-visible, externally-hosted).

**Fix**: local image_url constrained to `/tmp/wp_content/` prefix via `resolve()` check; also passes through `validatePathSafety(canonical, "read")` as belt-and-braces.

### C-SSRF-1 — wp_media_upload unvalidated URL fetch

**Evidence**: `src/tools/builtin/wordpress.ts:824` `await fetch(imageUrl, …)` with no `validateOutboundUrl`. SSRF to `http://169.254.169.254/`, localhost, internal services, etc.

**Fix**: added `validateOutboundUrl(imageUrl)` pre-flight.

### C-SSRF-2 — gemini_research unvalidated URL fetch

**Evidence**: `src/tools/builtin/gemini-research.ts:266` `fetch(source, …)` where `source` is LLM-controlled.

**Fix**: added `validateOutboundUrl(source)` pre-flight on URL branch.

## Round 2 — Additional Criticals (3)

Round 2 was performed by the `qa-auditor` subagent on the round-1 fixes. These were **missed entirely** by round 1 — the qa-auditor's independent review found them.

### C1 (R2) — pdf_read URL + local both unguarded

**Evidence**: `src/tools/builtin/pdf-read.ts:139-157` — URL branch calls `extractPdfFromUrl(source)` with no validateOutboundUrl; local branch calls `extractPdfToMarkdown(source)` with no validatePathSafety.

**Impact**: Same class as C-SSRF-2 + C-PATH-3 but a third tool entirely.

**Fix**: added both guards on the respective branches.

### C2 (R2) — gemini_research local branch still unguarded

**Evidence**: `src/tools/builtin/gemini-research.ts:298-306` — round-1 fix added guard to URL branch (line 266) but the `else` branch at 298 (`filePath = source`) + `readFileSync(filePath)` at 306 was left unguarded.

**Impact**: `source="/root/.claude/.credentials.json"` → uploads to Gemini.

**Fix**: added `validatePathSafety(source, "read")` on the local branch. **Lesson: when a tool has a URL-or-local dispatch, both branches need the corresponding guard; fixing only one is half a fix.**

### C3 (R2) — contentFile exfil across 4 tools

**Evidence**:

- `src/tools/builtin/google-docs.ts:494` (`google_docs_append_text`)
- `src/tools/builtin/google-docs.ts:603` (`google_docs_apply_revisions`)
- `src/tools/builtin/google-drive.ts:521` (`google_drive_create_or_update`)
- `src/tools/builtin/wordpress.ts:551` (`wp_publish`)

Each accepts an LLM-controlled `content_file` absolute path → `readFileSync(contentFile, "utf-8")` → publishes/appends as document/post body. Round 1 fixed wp_media_upload's media path but missed wp_publish's contentFile.

**Impact**: LLM sets `content_file="/root/.claude/.credentials.json"` and the tool uploads the credential as document body. Four separate external-publish surfaces.

**Fix**: `validatePathSafety(contentFile, "read")` added before each readFileSync.

## Round 1 — Majors (4)

### M-CTR-1 — nanoclaw mission-control RW mount

**Evidence**: `src/runners/nanoclaw-runner.ts:56` pre-fix: `/root/claude/mission-control:/root/claude/mission-control:rw`. nanoclaw-worker.ts only reads `dist/` and `package.json`.

**Fix**: flipped to `:ro`; tests at `nanoclaw-runner.test.ts` updated (2 assertions).

### M-SHELL-1 — shell_exec reader-bypass of READ_BLOCKED_PATHS

**Evidence**: `src/tools/builtin/shell.ts:20-45` DENY_COMMANDS blocks `rm/mkfs/dd/sqlite3/systemctl/...` but NOT `cat/head/tail/less/xxd/strings/awk/sed/grep/base64/md5sum/…`. With round-1's READ_BLOCKED_PATHS in place, the LLM could still do `shell_exec("cat /root/.claude/.credentials.json")` and get the credential in tool output.

**Impact**: The READ_BLOCKED_PATHS denylist was trivially bypassable on any runner with shell_exec.

**Fix**: added 6 new DENY_PATTERNS entries matching reader commands against each READ_BLOCKED_PATHS family (credentials.json, /root/.ssh/, /root/.gnupg+.aws+.docker+.kube+.config/gh, /etc/shadow+sudoers+ssh, /proc/self/environ+mem, /root/.npmrc+.netrc+.pgpass+.gitconfig+.git-credentials).

### M-RATE-1 — `/api/*` and `/a2a/*` unrated

**Evidence**: `src/api/index.ts:80` pre-fix only had `apiKeyAuth`; no rate-limit on `/api/*`. `/a2a/*` also unrated.

**Fix**: new `src/api/rate-limit.ts` (IP-keyed sliding window, 300 req/min). Wired to both `/api/*` (W3 surfaced extension to `/a2a/*`) **before** `apiKeyAuth` so pre-auth traffic also counts toward the limit (prevents API-key brute-force).

### M-CRED-FILTER — jarvis-dev SENSITIVE list too narrow

**Evidence**: `src/tools/builtin/jarvis-dev.ts:187` pre-fix: `[".env","credentials","secret",".key",".pem","token"]`. Missed `id_rsa`, `id_ed25519`, `.npmrc`, `.netrc`, `.gitconfig`, `.git-credentials`, `.pgpass`, `.p12`, `.pfx`, `.crt`.

**Fix**: expanded SENSITIVE to 18 entries covering private-key filenames + cert extensions + auth dotfiles.

## Round 2 — Additional Majors (3)

### M1 (R2) — shell_exec reader bypass (**promoted to round-2 fix**)

Surfaced by round 2; fix landed as M-SHELL-1 above. Listing here as a separate round-2 finding emphasizes that the READ_BLOCKED_PATHS denylist by itself was insufficient and the shell-path closure was necessary.

### M2 (R2) — Symlink escape around validatePathSafety

**Evidence**: `src/tools/builtin/immutable-core.ts:209` pre-R2 round 2 fix: `resolve(path)` only normalizes `.` / `..`; does NOT follow symlinks. Attack: `ln -s /root/.claude/.credentials.json /tmp/safe.txt` + `file_read("/tmp/safe.txt")` → resolve returns `/tmp/safe.txt`, denylist passes, readFileSync follows the symlink.

**Fix**: after `resolve()`, also `realpathSync(candidate)` (wrapped in try/catch for non-existent paths); denylist is checked against BOTH the resolved path AND the realpath. Live-tested: `/tmp/sec_probe_symlink -> /root/.claude/.credentials.json` now blocks with `'/root/.claude/.credentials.json' is a read-blocked secret file`.

### M3 (R2) — X-Forwarded-For rate-limit bypass

**Evidence**: `src/api/rate-limit.ts` pre-R2: `clientIp` read `X-Forwarded-For` first without checking whether the socket was localhost. Server binds `0.0.0.0:8080` (UFW-open). Attacker hitting `:8080` directly could rotate `X-Forwarded-For: 1.2.3.<n>` per request to scatter the bucket.

**Fix**: XFF is only trusted when `TRUST_FORWARDED_FOR=true` env is set AND the socket remote address is localhost (`127.0.0.1` / `::1` / `::ffff:127.0.0.1` — the Caddy reverse-proxy topology). Otherwise we key on the socket `remoteAddress` directly. Also added `MAX_TRACKED_IPS = 10_000` wholesale-eviction cap (W4).

## Key patterns captured

1. **Double-audit is load-bearing for security dimension.** Round 1 closed 6 Criticals; round 2 surfaced 3 more Criticals on tools round 1 hadn't even looked at. Single-round security audits ship undiscovered bugs. The methodology's double-audit requirement for Security + Resilience is not a formality.

2. **Fixing one branch of a dual-dispatch tool is half a fix.** C-SSRF-2 (round 1) fixed gemini_research's URL branch; C2 (round 2) found the local branch still unguarded on the same tool. Whenever a tool has a URL-or-local dispatch, both branches need the corresponding guard — grep the handler for both `.startsWith("http")` _and_ the else-clause.

3. **Denylists need realpath + symlink-resolution.** `resolve()` only normalizes `.` / `..`; an LLM with write access to `/tmp/` can create a symlink to the target and bypass any string-match denylist. Every path denylist must run both `resolve()` and `realpathSync()` and check both.

4. **Process-level guards must match tool-level guards.** READ_BLOCKED_PATHS in file.ts was trivially bypassable via `shell_exec("cat ...")`. Any new tool-level guard needs a symmetric shell_exec DENY_PATTERN. Treat shell_exec as a universal-escape-hatch that requires matching guards for every tool-level defense.

5. **`contentFile` is an exfil pattern across the codebase.** Any tool that accepts `content_file` / `contentFile` / `source` / `image_url` as an LLM-controlled absolute path and then `readFileSync`s it is an exfil vector if the tool ships its payload externally (WordPress, Google Docs, Google Drive, Gemini). Audit cadence: every new external-publish tool gets a mandatory readFile-path-allowlist review.

6. **XFF must not be trusted when the server is directly exposed.** Unless the socket remote address is localhost AND a trust-opt-in env is set, `X-Forwarded-For` is attacker-controlled and must be ignored. Per-IP rate-limits that trust XFF by default are defense-in-paper.

7. **Sanitization paths must be wired on every runtime branch.** `sanitizeToolResult` lived in adapter.ts (OpenAI path) for months; when claude-sdk became primary, the sanitization was silently left behind. Every security function needs a grep — `where is it called from?` — whenever the primary runtime changes. Matches the `feedback_incomplete_migration.md` pattern: "X is now primary" touching one file = partial migration.
