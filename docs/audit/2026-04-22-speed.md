# Dimension 2 — Speed audit

> **Status**: COMPLETE — 8 probes run, 1 fix landed live, 6 findings documented, 4 deferred with triggers.
> **Baseline**: `../benchmarks/2026-04-22-baseline.md`
> **Methodology**: `../planning/stabilization/full-system-audit.md`
> **Post-fix commit**: see Commits section below.
>
> **Headline**: fast-runner p90 is **122s vs 20s target** (6×). The single biggest wall-clock leak is **Hindsight recall timing out at exactly 5000ms on 100% of calls** — Hindsight's agentic recall empirically takes ~9s, Jarvis's 5s timeout fires every time, SQLite fallback completes in ~300ms, so users pay 5.3s per recall instead of 300ms. Shortened client timeout 5000ms → 1500ms; verified live (5005ms → 1505ms per failed recall). Heavy-runner PER phase timing was incorrectly described as "unobservable" in methodology — `runs.trace` JSON blobs already persist per-phase start/end timestamps; 7d p50: plan 28.5s, execute 190s, reflect 25.3s.

### Finding

**Critical — fast-runner p90 is 6× slower than target (122s vs ≤20s).**

- Memory recall stalls: until this audit, `memory_search`-via-tool calls blocked for 5s each on Hindsight timeouts (see S7). Sonnet's default behavior is to call context tools early.
- Per-turn conversation-history growth with no upper bound on hot tool loops.

Without **per-turn instrumentation** — LLM-call vs tool-call vs memory-call breakdown — the root cause is speculative.

### Decision — **Defer** (instrumentation-gated)

No ship-today fix identified. Follow-ups:

- **T-Inst-1** (trigger: next audit cycle) — add per-turn timing in fast-runner emitting `[fast-runner] turn N: llm=Xms tools=Yms memory=Zms total=Tms` so the 122s p90 can be decomposed. Blocks a targeted fix.

## S2 — Heavy-runner PER loop wall-clock

### Measurement

Methodology said to use `task_provenance` phase timestamps. That table is effectively dead (0 rows since 2026-04-14; 76 rows lifetime). HOWEVER `runs.trace` persists the Prometheus orchestrator's JSON trace including `phase_start` / `phase_end` events per phase.

**7d heavy-runner PER phase breakdown (n=7 tasks):**

| phase   | n   | p50        | p90        | max        |
| ------- | --- | ---------- | ---------- | ---------- |
| plan    | 7   | 28,535 ms  | 34,922 ms  | 34,922 ms  |
| execute | 7   | 190,138 ms | 256,741 ms | 256,741 ms |
| reflect | 7   | 25,330 ms  | 44,532 ms  | 44,532 ms  |

**Total end-to-end 7d:** p50 ≈ 244s, p90 ≈ 336s, max 321s observed.

### Finding

**Info — methodology's "unobservable" assumption was wrong; PER phase data is in `runs.trace`.**

- Execute phase is 6-7× longer than plan or reflect — consistent with it being the only multi-turn multi-tool phase.
- Plan alone at 28.5s p50 is significant: one LLM call + goal-graph building. Under Sonnet with large prompt + deferred-tool loading it's plausible, but worth tracking.
- Methodology target "heavy-runner PER p90 ≤45s for trivial task" is impossible even in theory: plan (28.5s) + reflect (25.3s) = 53.8s of unavoidable structural cost at current model latencies, before execute does a single thing. Either the target needs revision or plan/reflect need parallelization.

### Decision — **Defer** (tracker)

- **T-Inst-2** (trigger: any p90 heavy task >5min) — surface `phase_durations` in task dashboard pulled from `runs.trace`. Mechanical; aligned with `mc-ctl` stats.
- **T-Goal-1** (trigger: post-freeze roadmap) — revisit heavy-runner PER loop target. 45s p90 assumed fast model ~10s per phase; Sonnet reality is ~25-30s per phase.

## S7 — Thread-buffer / memory recall latency

### Measurement

Direct Hindsight recall probe (`/v1/default/banks/mc-jarvis/memories/recall`, `budget: "low"`):

```
probe 1: real 0m8.948s
probe 2: real 0m9.186s
probe 3: real 0m9.132s
```

Meanwhile Hindsight `GET /health` returns in 3ms. Container is up; the agentic recall path itself is slow because Hindsight uses an LLM agent (`openai/qwen3.5-plus` per `docker logs crm-hindsight`) to reason over memories. Observed Hindsight mental-model refresh jobs logging `llm=[agent_1_err=36106ms, agent_2_err=35628ms, final=19660ms] (91394ms)`.

Jarvis's Hindsight client timeout was **5000ms** (`src/memory/hindsight-client.ts:14`). Every recall call timed out at ~5005-5040ms, triggered the SQLite fallback (~300ms), and the user paid **5.3s wall-clock per recall**.

Verified in journalctl for the session prior to this audit:

```
Apr 22 20:07:43 ... [memory] recall(hindsight) FAILED after 5022ms — falling back to SQLite
Apr 22 20:07:43 ... [memory] recall(hindsight) FAILED after 5037ms — falling back to SQLite
Apr 22 20:07:43 ... [memory] recall(sqlite-fallback) results=3 276ms
Apr 22 20:07:44 ... [memory] recall(sqlite-fallback) results=5 412ms
```

100% timeout rate. Circuit breaker (threshold=3, 60s cooldown) DID NOT trip in this window — only 2 calls happened before the process quiesced, so the fast-runner traffic profile pays full 5s every ≥60s window.

### Finding

**Critical — 5000ms timeout × 100% Hindsight failure rate = 4700ms of wall-clock waste per memory recall.**

Measured impact on end-to-end fast-runner latency:

- Fast-runner tasks that call `memory_search` via tool (or hit any code path that calls `getMemoryService().recall()` — planner, executor, MCP `memory_search` tool) pay 5.3s per call.
- Hindsight's 9s agentic recall is an upstream-service tuning issue; Jarvis can't fix Hindsight's LLM backend from here.

### Decision — **Fix** (landed, verified live)

- `src/memory/hindsight-client.ts` — introduce `RECALL_TIMEOUT_MS = 1500` constant; recall path uses it instead of `DEFAULT_TIMEOUT_MS = 5000`. Retain and reflect paths still use 5000ms (non-user-interactive).
- `src/memory/hindsight-backend.ts` — docstring updated to reflect new timeout + audit reference.
- `src/memory/hindsight-client.test.ts` — new test verifies recall aborts after 1500ms via `AbortSignal`.

**Live verification (3 back-to-back probes against real Hindsight post-deploy):**

```
RECALL FAILED 1506ms: This operation was aborted
RECALL FAILED 1505ms: This operation was aborted
RECALL FAILED 1505ms: This operation was aborted
```

**Per-recall wall-clock savings: 5005 → 1505 ms ≈ 3500 ms (~70%)** before SQLite fallback runs.

Circuit breaker threshold unchanged (3 failures) — warm-up cost is now 3 × 1.5s = 4.5s (was 3 × 5s = 15s). Once circuit opens, recalls go straight to SQLite (~300ms) for 60s. Half-open probe costs 1.5s per minute (worst case), was 5s.

## S8 — Scheduled-task polling loop query plans

### Measurement

Dynamic scheduler uses `cron.schedule("* * * * *", ...)` (once per minute). Inside the callback, it calls `listSchedules(true)` which runs:

```sql
SELECT * FROM scheduled_tasks WHERE active = 1 ORDER BY created_at ASC
```

**EXPLAIN QUERY PLAN:**

```
SCAN scheduled_tasks
USE TEMP B-TREE FOR ORDER BY
```

Full-table scan + in-memory sort. With **8 total rows (7 active)**, this is microseconds of CPU per minute.

The schedule_id CRUD query `WHERE schedule_id = ?` correctly uses `sqlite_autoindex_scheduled_tasks_1` — the unique index on schedule_id.

### Finding

**Info — no performance issue at current scale.**

The SCAN + TEMP B-TREE is not "bad" when n=8. Would not need to optimize unless schedule count grows past ~10,000.

### Decision — **No action**

Optional cleanup (not landing): the ORDER BY is semantically unnecessary for the poll loop — schedules within a single minute are interchangeable. Removing `ORDER BY created_at ASC` would drop the temp B-tree but save ~0 ms. Not worth a commit.

## Key patterns captured

1. **Timeout must align with dependency reality.** Jarvis's 5s Hindsight timeout was set when Hindsight was presumed to be a fast vector DB. It became an agentic LLM-backed service (~9s per query) upstream; Jarvis's client timeout never followed. Any upstream dependency's actual latency profile must be re-probed whenever its implementation shifts from "fast infra" to "LLM-backed."
2. **"Unobservable" is sometimes a methodology error.** S2 assumed PER phase timings weren't accessible; they were — in `runs.trace` — just never surfaced. Always grep the codebase for the named entity (`phase_start`, `phase_end`, `trace`) before declaring no data.
3. **In-memory-only metrics are audit traps.** S4, S5, S6 all have the same problem: the instrumentation exists but is erased on restart and capped to top-N. Restart-surviving metrics are prerequisite for any speed audit worth its name.
4. **Phase cost floor makes aggregate targets implausible.** Heavy-runner "≤45s for trivial task" was written assuming ~10s/phase. At Sonnet's ~25-30s/phase, the floor is already 53s for plan + reflect alone. Either parallelize (plan concurrently with pre-loading), prune a phase, or revise the target.
5. **Single-digit-row scans are not optimization targets.** S8's full-table SCAN looked like a finding until the row count (8) rendered the SCAN microseconds. Always measure before proposing an index.
