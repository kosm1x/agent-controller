# v7.6 Spine 4 — MCP Annotation Coverage Audit

**Phase**: v7.6 reliability (`docs/V7.6-GUIDE.md`)
**Spine**: 4 — MCP annotation coverage push
**Started/closed**: 2026-05-08 (single session)
**Lens**: every production-registered Tool gets all 4 MCP hints (`readOnlyHint` / `destructiveHint` / `idempotentHint` / `openWorldHint`) explicitly. No silent fallback to `getToolAnnotations` defaults.

## Mission

Pre-spine baseline: 23 of 185 builtin Tool definitions had any annotations (12.4%). The other 162 silently relied on `getToolAnnotations` conservative defaults: `{readOnly:false, destructive:true, idempotent:false, openWorld:true}`. Every audit-claim metric segmenting on destructive-vs-readonly was muted by this default-collapse.

Goal per V7.6-GUIDE.md: every production tool annotated, with registry-level invariant tests proving no defaults leak.

## Approach

1. **Inventory script** — walked `src/tools/builtin/*.ts`, identified every `export const xxxTool: Tool = {`, classified each by hint coverage.
2. **Bulk annotation script** — for each unannotated tool, looked up the variable name in a per-tool classification table and inserted the 4 hints right after the `name:` line. Brace-tracking edge cases (template-literal descriptions with `${...}` interpolations) caused 1 false negative (`diagramGenerateTool`) which was fixed manually.
3. **Registry-level invariants** — added new `describe("MCP annotation coverage (v7.6 Spine 4)")` block in `registry.test.ts` asserting:
   - Every production tool has all 4 hints explicitly set (no `tool[hint] === undefined`)
   - `readOnlyHint=true` ⇒ `destructiveHint=false` (per type-def comment)
   - `requiresConfirmation=true` ⇒ `readOnlyHint!=true` (per CLAUDE.md)
   - `riskTier='high'` ⇒ `destructiveHint=true` (consistency)
   - `getToolAnnotations` consumer-side check: no field falls back to default
   - Coverage statistic: `ALL_TOOLS.length >= 180` (sanity threshold)

## Annotation classification rules

Per-tool classification was driven by the tool name + a brief inspection of its handler. Four canonical hint sets:

| Code | readOnly | destructive | idempotent | openWorld | Use for                                                     |
| ---- | -------- | ----------- | ---------- | --------- | ----------------------------------------------------------- |
| `r`  | true     | false       | true       | true      | Pure reads against external state (web search, market read) |
| `rL` | true     | false       | true       | false     | Pure local compute / pure transform (batch decompose)       |
| `d`  | false    | true        | false      | true      | State-mutating action (write, send, publish, delete)        |
| `di` | false    | true        | true       | true      | Idempotent mutation (file_write same content = no diff)     |

**Key calls**:

- **Read tools that hit external APIs** (web search, market quote, geocoding) → `idempotent:true` because they have no SYSTEM state effect even though external state may change
- **Local compute tools** (currency convert via cached rates, batch decompose) → `openWorld:false` if they don't hit network/FS/DB
- **File write with same content** → `idempotent:true` (`file_write`, `jarvis_file_write`)
- **Sends/publishes/inserts** → `idempotent:false` (each call creates a new artifact)
- **`requiresConfirmation:true` tools** never get `readOnlyHint:true`; the test enforces this
- **Database-write tools** (`schedule_task`, `user_fact_set`, etc.) → `openWorld:true` (DB is "outside the agent's bounded environment" per the type def)

## What this enables

- **`mc-ctl audit-claim` segmentation by destructive-vs-readonly** is now meaningful — the destructive cohort is the actually-destructive cohort, not "unannotated default-collapse".
- **MCP-spec compatibility** — when the agent surface exposes tools to MCP clients (in v7.6+), the hints transmit as authoritative metadata.
- **Future-tool gate** — registry test trips immediately on any new tool that forgets to set all 4 hints. Forces the conversation at PR time, not weeks later.

## What this does NOT do

- **MCP source dynamic schemas** — tools registered at runtime by the MCP source (xpoz, browser, etc.) get their hints from the upstream definition, not our codebase. Out of scope for this spine.
- **idempotent precision** — for tools where idempotency depends on the exact arg values (e.g., `gsheets_write` is idempotent if you write the same value, not idempotent if you append), the hint is conservative ("often not idempotent"). Could be refined with arg-aware annotations in a future spine.
- **External tool registry sync** — the `mc-ctl tool-list` output doesn't yet display the hints; cosmetic addition.

## Round 1 — qa-auditor on `331678b` (PASS WITH WARNINGS → 3 fixes shipped)

Audit found 5 issues (W1+W2 classification errors, W3 stale numbers, W4 MCP source residual, W5 borderline screenshot classification). 3 of 5 are addressed in this round; 2 deferred with explicit notes.

**W1 (CRITICAL bundle-regression) — `batchDecomposeTool` mis-classified as `rL`** (`batch.ts:14-17`)

- Pre-fix: `readOnly:true, destructive:false, idempotent:true, openWorld:false`. The handler calls `submitTask()` (line 112) which mutates the dispatcher task queue (DB write). Re-issuing the same call enqueues N MORE subtasks — explicitly NOT idempotent.
- **Fix**: re-classified to `d` (`readOnly:false, destructive:true, idempotent:false, openWorld:true`). Inline comment names the audit finding.
- Same class as Spine 1 round-1 C1 — bundle-regression caught at round-1 saves shipping wrong metric data.

**W2 (MEDIUM bundle-regression) — `humanizeTextTool` mis-classified as `d`** (`writing.ts:40-43`)

- Pre-fix: `readOnly:false, destructive:true, idempotent:false, openWorld:true`. Pure LLM transform — no FS write, no DB write, no external state change. Returns transformed text only.
- **Fix**: re-classified to `r` (`readOnly:true, destructive:false, idempotent:true, openWorld:true`). Inline comment names the audit finding.
- Would have polluted the destructive-cohort metric the spine claims to enable.

**W3 (test depth) — coverage threshold loose** (`registry.test.ts:367` pre-fix)

- Pre-fix: `expect(ALL_TOOLS.length).toBeGreaterThanOrEqual(180)` allowed up to 6 tools to silently regress out of the registry. Audit doc reported 184; actual count 186.
- **Fix**: pinned to exact `expect(ALL_TOOLS.length).toBe(186)` with comment naming the forcing-function. Audit doc count corrected.

**W4 (DEFERRED) — MCP source residual is bigger than the doc admitted**

- The MCP source registers tools at runtime (xpoz, browser, playwright). Their hints come from upstream definitions, not our codebase. Tools registered through it fall back to `getToolAnnotations` defaults — exactly the metric-muting problem this spine claims to close.
- **Status**: documented as residual. Out-of-scope for this bundle but tracked: `mc-ctl audit-claim` segmentation queries should filter `tool LIKE 'mcp_%'` until upstream MCP servers ship hints with their schemas. Adding a wrapper `getMcpToolAnnotations()` that overrides hints by tool-name pattern is a candidate for a v7.7 follow-up.

**W5 (DEFERRED) — `screenshotElementTool` annotation is borderline**

- Annotated `d` (destructive). Writes a timestamped PNG to `/tmp/screenshots/` — never overwrites, but accumulates. By strict MCP-spec semantics ("destructive = MAY perform irreversible action"), `/tmp` accumulation is reversible. By a wider read ("destructive of disk space when called repeatedly"), `d` fits.
- **Status**: kept as `d` (conservative — a tool that fills `/tmp` is operationally destructive even if individually reversible). Re-evaluate if a 5th canonical hint set (`s` = side-effecting but reversible) is introduced in a future spine.

## Lessons

1. **Script-driven mass edits need string-aware parsers.** The brace-tracking failed on Tool definitions with template-literal descriptions containing `${...}` interpolations. Cost: 1 missed tool (manually fixed) + 24 false negatives in the inventory output that wasted ~10 min of debug time before re-inventory with string-aware parser.
2. **The "23 already annotated" count from earlier sessions correlates exactly with high-risk tools** — file/jarvis-files/shell/gmail/web-read/web-search/code-search/http. Operators annotated the dangerous ones first; the long tail is read tools and helpers.
3. **Coverage threshold tests need a generous floor**. The first attempt asserted `>= 180` but `ALL_TOOLS.length` was 158 (Google + Memory + Skills tools live in `builtin/*.ts` but get registered through their own ToolSource adapter, so they weren't in `BUILTIN_TOOLS`). Fix: extend ALL_TOOLS to include those tool exports directly. Same lesson as the Spine 2 `getAllAvailableTools` test threshold.
