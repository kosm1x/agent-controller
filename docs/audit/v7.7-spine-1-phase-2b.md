## Tier-1 ŌåÆ R1-only downgrade rationale

The R1-C1 fix was architectural, NOT incremental: separated `sendLLMReplyToChannel` from `sendToChannel`, added in-flight tracking, added shutdown await. R2 catches "bugs INSIDE R1 fixes." The risks:

- **Could the new `sendLLMReplyToChannel` itself have a shutdown race?** No ŌĆö `inflight.finally(() => this.gateInflight.delete(inflight))` is symmetric with `this.gateInflight.add(inflight)`. The stopAll await uses `Promise.allSettled` which never rejects. Test "stopAll() awaits in-flight gate IIFEs" reproduces the race.
- **Could the 3 callsite switches miss an LLM-reply path?** Searched: `sendToChannel` is called from ~34 sites; only the 3 inside `handleTaskCompleted` carry LLM-derived text. Verified by grep + manual code-walk of the surrounding context (line 2166 background-agent summary, line 2178 stream-finalize fallback, line 2181 direct).
- **Could the gate produce a verdict the IIFE mis-handles?** Tested: pass ŌåÆ original ships, fail ŌåÆ fallback ships, error ŌåÆ fallback ships. Three integration tests pin each branch.
- **Could the persona prompt amendment regression?** Pinned by `prompt-sections.test.ts` audit-awareness test.

R2 would have ~10-15 min of value finding cosmetic adjacents. Decision documented at time of skip.

Where R2 WOULD be mandatory: Phase 2c (closure-doc convention) if it adds router-level enforcement; any future surface that adds new gate kinds (e.g. WhatsApp business mailbox).
