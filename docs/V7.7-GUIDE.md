## Anti-mission — what v7.7 will NOT do

- **No V8.1 / V8.2 / V8.3 capability work.** Substrate only. If a substrate item creeps into delivering capability behavior, defer the capability and ship just the substrate primitive.
- **No new tools.** 248 is enough. S5 _registers_ skills as first-class capabilities; it does not author new ones.
- **No new runners.** Five is enough.
- **No memory architecture changes.** pgvector + Ebbinghaus + unified FS + kb_entries layer hierarchy already work. Don't redesign them.
- **No substrate scout adoption mid-phase.** datagen/voyager/letta/smolagents/pheropath stay reference-only; v7.7 composes patterns from them, does not adopt the libraries.
- **Operator-profile: ⚠️ SKELETON ONLY** (Q3 carve-out from original anti-mission). Storage shape + read API; NO inference / NO personality / NO preference / NO write-side population beyond what Conway Pattern 2 cohort roll-up already produces. V8.2 owns the semantic layer. **Anti-mission compliance test**: can V8.2 redefine the value-side semantics without migrating data? If yes, the scaffold is thin enough. If no, revert.
- **No closure-ritual automation as a feature** — wait, this one is carved out too (Q4 absorption). Spine 7 Part B codifies `mc close-phase` / `mc resume` / `mc audit-closure`. Constraint: tools MUST NOT execute git ops (commit/tag/push) without operator confirmation.

If a candidate item feels like a v8 capability, **it is**. Defer.

### Spine 4 — Conway Pattern 1: General-events middle layer

**Reference**: `~/.claude/projects/-root-claude/memory/reference_conway_2005_sms.md`

**Why it's substrate**: V8.1 spec explicitly cites Pattern 1 as a structural prerequisite. Today, episodic memories live as raw events in the journal/cost_ledger; semantic memories live as facts in jarvis_files. There's no middle layer that aggregates events into "what's been happening this week" form.

**Shape**: derive a `general_events` view (or table) over journal + cost_ledger + recall_audit + decision_events. Time-windowed aggregations: per-day, per-week, per-project. Inputs to V8.1's morning-brief proactive scan.

**Done when**: V8.1's Phase 1 spec can run against this layer without further build-out.

### Spine 5 — Conway Pattern 2: Self-defining cohort + ⚠️ operator-profile skeleton (Q3)

**Reference**: `reference_conway_2005_sms.md`

**Why it's substrate**: per V8-VISION §4, Pattern 2 is V8.2 prerequisite per Conway's empirical claim ("V8.1 reads as generic without it"). The system needs a stable model of _which projects, which contacts, which threads_ are the operator's identifiable cohort, so morning-brief judgments can speak in those terms instead of generic "you have 3 stalled tasks" language.

**Shape**: derive cohort membership from existing data — NorthStar projects, frequent CRM contacts, recurring conversation threads. Daily roll-up. No new schema if existing data supports it; one small denormalized table if not.

**Plus (Q3 resolution 2026-05-18) — operator-profile skeleton**: schema-only `operator_profile` table with `(cohort_member_id, attribute_key, attribute_value, written_at, written_by)`. Write-side populated ONLY by Conway Pattern 2 cohort roll-up. NO inference, NO LLM-derived fields. V8.2 owns what `attribute_key`/`attribute_value` mean semantically. v7.7's contract is just "the bone structure exists and V8.2 can extend without migration."

**Anti-mission compliance test for the skeleton**: can V8.2 redefine value-side semantics without data migration? If yes, the skeleton is thin enough. If no, revert.

**Done when**: V8.1's Phase 2-3 (where cohort framing kicks in) can compose against the cohort definition without further design. AND operator-profile skeleton table shipped + write-path test that it never gets populated except by cohort roll-up.

### Spine 6 — Conway Pattern 3: Named recall modes + correspondence audit

**Reference**: `reference_conway_2005_sms.md` + V8-VISION §9 trap warnings

**Why it's substrate**: V8-VISION §9 explicitly names "coherence drift toward confabulation" as a top trap. Pattern 3 is the codified defense: named recall modes (coherence vs correspondence) and a periodic correspondence-audit task. Without it, V8.2's evidence-cited proposals will drift confidently-wrong over months.

**Shape**: tag every recall result with mode (`coherence` = pattern-match historical, `correspondence` = check-current-state). Weekly correspondence-audit ritual that re-validates a sample of recently-cited memories against current reality. Surfaces stale memories before they end up in V8.2 proposals.

**Done when**: `recall_audit.mode` column live, weekly correspondence-audit cron emitting findings to `drift_signals` (composes with Spine 2). Stale-rate baseline measured.

## Operating rules

All v7.6 rules carry forward:

1. **Instrument first, decide later** (perennial)
2. **Audit every bundle** — qa-auditor pass on every ship-it. By v7.7 close, the 33% real-bug rate is a calibrated fact across 3 phases.
3. **Catalog-then-execute** for multi-domain backlogs
4. **Risk carve-outs everywhere** — every ergonomic-relief gate has a high-risk override
5. **Producer/consumer coupling tests** — every observability surface has both ends pinned in one test
6. **Closure ritual every phase**
7. **No new features mid-phase** — extends to "no v8 capabilities mid-substrate"

**New for v7.7**:

8. **Spec-first delta tracking** — for each spine, the deliverable diff against the existing spec is itself an artifact. Land `docs/planning/v8-substrate-s{2,3,5}-impl.md` files that record what shipped vs what specced. Makes V8-VISION §10 updates trivially auditable.
9. **Q3 carve-out vigilance** — every PR touching `operator_profile`-related code gets reviewed against the anti-mission compliance test (can V8.2 redefine value-side semantics without migration?). If no, REJECT — that's V8.2 surface, not v7.7 skeleton.
10. **Continuity-tooling git-op gate** (Q4 absorption) — every git-op codepath in `mc close-phase` / `mc audit-closure` has an explicit "operator confirmed?" gate. Test: try to invoke without confirmation; tool must abort with no side effects.

## What we're betting on

v7.7's bet is that **shipping the substrate v8 already named is more honest than building new pillars**. The 4 deltas resolved 2026-05-18 (S5 all-5, Conway all-3, operator-profile skeleton, continuity tooling absorption) make this a "go wide" phase: every prerequisite V8.1/V8.2/V8.3 will need is on the docket, plus the continuity bone V8 will run on, plus the ritual tooling that makes phase boundaries durable.

If at v7.7 close we've:

- Shipped S2 + S3 + S5 (all 5 phases) + Conway 1-3 + operator-profile skeleton
- Shipped continuity tooling and used it for v7.7's own closure
- Updated V8-VISION §10 to read ✅ on the foundation row
- Started zero new capabilities

…then **V8.1 has clean ground to land on**, V8.2 has its bone structure pre-poured, and v7.7 has been a successful substrate phase.

The temptation in week 4 will be "we're so close to V8.1, let's just start Phase 1." Resist. Close the substrate phase explicitly. The closure ritual exists for a reason — premature pivots leak debt.
