## What this spine is

**Conway Pattern 3 — Named recall modes + a periodic correspondence audit.**

Conway's empirical warning (`reference_conway_2005_sms.md` §Pattern 3):
pure-**coherence** recall (filtering out memories that contradict the
"competent system" self-image) drifts into **confabulation**. V8-VISION §9
names exactly this — "coherence drift toward confabulation" — as a top v8
trap. Pattern 3 is the codified defense.

Today, Session 117's outcome filter (`excludeOutcomes` — excludes
`failed`/`concerns` memories by default) is an _unnamed_ coherence move.
Spine 6 names the modes and adds the audit that keeps coherence honest.

- **`coherence`** (default) — excludes failed/concerns; used by V8.1 briefs, V8.2 proposals (forward-looking, goal-supportive).
- **`correspondence`** (explicit) — includes everything; used by retrospective audits, S2 critic verification, post-mortems.
- **`unfiltered`** (debug) — `mc-ctl recall` paths.

Plus a **weekly correspondence audit**: query in correspondence mode, surface what coherence-mode filtering has been suppressing, and alert if a true-failure memory with high relevance keeps getting filtered out.

## Shape (from V7.7-GUIDE §"Spine 6")

1. **`recall_audit.mode` column** — tag every recall result with its mode. Additive `ALTER TABLE ADD COLUMN` (not a CHECK change → applies live, no DB reset).
2. **Rename the recall option** — Session 117's `excludeOutcomes: string[]` default becomes `recall_mode: 'coherence' | 'correspondence' | 'unfiltered'`. This touches recall callers — handle the API change carefully (keep `coherence` the default so behavior is unchanged).
3. **Weekly correspondence-audit cron** — re-validates a sample of recently-cited memories against current reality; emits findings to **`drift_signals`** (composes with Spine 2's S3 substrate — the audit is a drift signal source).
4. **Stale-rate baseline** — measure and record the baseline stale-rate so future audits have a reference.

**Done when**: `recall_audit.mode` column live; the weekly correspondence-audit cron runs and emits to `drift_signals`; the stale-rate baseline is measured and documented.

## Anti-mission reminder

- ❌ Don't build V8.1/V8.2 capability. Spine 6 ships the recall-mode tagging + the audit cron — substrate.
- ❌ Don't redesign the recall stack. The `excludeOutcomes` → `recall_mode` change is a rename + enum, not a re-architecture.
- ❌ Don't over-scope the correspondence audit. Sample + re-validate + emit a drift signal. No LLM-heavy re-reasoning.

## Operating rules that especially apply

1. **Audit every bundle** — qa-auditor R1 per ship; R2 only on Critical-bearing/contract-drift bundles (0 R2 across Spines 4 + 5).
2. **Schema discipline** — `ALTER TABLE ADD COLUMN` is additive and live-safe; a CHECK-enum change is not (never reset the DB). If `recall_audit.mode` wants a CHECK, ship the column without one or accept the enum is app-enforced.
3. **API-rename care** — `excludeOutcomes` → `recall_mode` touches every recall caller. Grep them all first; keep `coherence` the default so the rename is behavior-neutral. Cascading type changes break silently — assert the new field in existing tests.
4. **Compose, don't duplicate** — the correspondence audit emits to Spine 2's `drift_signals`; reuse that substrate, don't build a parallel one.
