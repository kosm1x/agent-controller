## What this phase is

**S5 Phase 4 — Callable-tools mode** — graduate skills from a retrieval surface into a CALLABLE one. Today the runner can VIEW (`retrieveSkills`) and TEST (`runSkillTests`) skills; Phase 4 lets it INVOKE them.

**Spec sections to read first**: `docs/planning/v8-substrate-s5-spec.md` §7 (lifecycle tools), §11 (ToolSource integration), §15 Q2 (script sandbox), §15 Q4 (cross-skill composition). Reading order: §7 → §11 → §15 Q2/Q4.

**Estimate**: ~2 days per spec §13; realistic 3-4h focused (matches recent bundle cadence). 2-3 bundles.

## Already shipped (do NOT rebuild)

- **Schema** (Phase 1): `skill_versions` / `skill_test_runs` / `skill_failures` tables + 10 added columns on `skills`
- **Critic + lifecycle** (Phase 2 B1): `runSkillCritic`, `skillSave`, operator override path
- **Test harness + sweep + skill-health CLI** (Phase 2 B2)
- **Vector retrieval** (Phase 3 B2): `retrieveSkills(taskContext, options)` returns ranked candidates with anti-list filtering + keyword fallback
- **`mc-ctl skills`** subcommand: list / backfill-embeddings / retrieve

What's **missing**: the dispatch surface (`skill_run`), tool-registration via ToolSource (top-k retrieved skills surface as deferred `skill__<name>` tools), and the per-skill cost-ledger wiring (`agent_type='skill:<name>'`).

## Anti-mission reminder

> Phase 4 REGISTERS skills as tools; it does NOT author the skill bodies.

If a bundle introduces a new skill BODY (markdown body of an actual SKILL.md), reject. The 5 production skills targeted at activation in Phase 5 are:

1. HTML compose suite (4 sub-skills already shipped in v7.6's `src/skills/catalog.ts` — these are FIRST-PARTY catalog skills, separate substrate)
2. Plus 4 from current rituals — TBD by operator; Phase 5 picks them, NOT Phase 4

Phase 4 only adds the dispatch surface. Bundle reviews must ask: "Does this PR contain skill instructions/steps that an LLM would execute, or only the tool surface that LOADS instructions from `skill_versions.body`?"

## Bundle plan (suggested — adjust as you learn)

### Bundle 1: skill_run + skill_load + skill_describe (~2h)

Per spec §7:

- `src/skills/dispatcher.ts` — `runSkill(name, args, options): SkillRunResult`. Looks up active version, validates `args` against `inputs_json` Zod-generated schema, executes via existing fast-runner with the skill body as system prompt + args JSON as user. Records to `cost_ledger` with `agent_type='skill:<name>'`.
- Wraps `runSkillTests`'s mini-runner pattern from Phase 2 B2 (which already does most of this — REUSE, don't duplicate)
- `src/tools/builtin/skill-tools.ts` — three new builtin tools: `skill_describe`, `skill_load`, `skill_run`. All `deferred: true`. Wire into ToolSourceManager.
- On failure: increment `skills.consecutive_failures`, INSERT into `skill_failures`
- On success: reset `consecutive_failures=0`, increment `success_count`
- Tests: validation error, tool unavailable, output mismatch (post-success path)

### Bundle 2: ToolSource auto-registration of top-k retrieved skills (~1.5h)

Per spec §11 Mode 2:

- When scope `skills` is active, the top-k matched skills from `retrieveSkills` register as deferred `skill__<name>` tools dynamically
- Tool descriptions = L1 description; schemas generated from `inputs_json` via dynamic Zod
- Hook into existing scope.ts / ToolSourceManager
- Tests: deferred registration, scope-gated, dynamic schema validation

### Bundle 3: cycle detection + observability (~1h, optional)

Per spec §15 Q4:

- `skill_run` from within a skill body invokes another skill → cycle detect (visited-set or depth counter, max 3)
- Cost-ledger query: `SELECT name, AVG(cost_usd), COUNT(*) FROM cost_ledger WHERE agent_type LIKE 'skill:%' GROUP BY agent_type`
- Prom counter `mc_skills_run_total{name, result}` with name-allowlist (closed cardinality)

## Risks specific to this phase

| Risk                                                                                | Severity | Mitigation                                                                                                                                              |
| ----------------------------------------------------------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `skill_run` becomes a recursive bomb (skill A calls B calls A)                      | High     | Cycle detection (visited set keyed by skillId) + depth bound = 3 (spec §15 Q4). Test with synthetic mutually-recursive skills.                          |
| Adversarial skill body steers the runner (same as Phase 2 B2's RUNNER_PREFIX issue) | High     | Reuse Phase 2's `RUNNER_PREFIX` pattern — first-instruction-wins discipline. The runner harness is shared.                                              |
| Cost-ledger row explosion (one row per skill_run)                                   | Medium   | Closed `agent_type` cardinality (one bucket per active skill); operator can purge old `cost_ledger` rows via existing tools                             |
| ToolSource registration races with retrieveSkills (top-k changes turn-by-turn)      | Medium   | Register on enrichment pass per turn; tear down on next enrichment. Or: stable allow-list of ALL certified skills (Phase 5 will have ≥5). Decide in B2. |
| `inputs_json` Zod-from-JSON generation is fragile                                   | Medium   | Constrain accepted shapes at Phase 1's parser; Phase 4 just consumes. Add tests for malformed inputs_json producing a clear validation error.           |
