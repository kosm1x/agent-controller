## Phase 1 — Harness + JSON schema enforcement (~2-3 days)

**Goal**: tool-boundary schema rejects any `report` tool output that lacks the typed evidence fields.

**Required fields** (per spec §2 Primitive A — DATAGEN's note-agent pattern):

- `verified_against`: array of data-source citations (see spec §3 for citation shape)
- `sample_n`: integer (the N behind any aggregate claim)
- `window`: ISO timestamp range (start, end)
- `claims`: array of `{statement, evidence_index}` — every claim must point into `verified_against`
- `concerns`: optional array of `{type, detail}` — known limitations, drift, low-confidence claims

**Mechanism**: Zod schema validation at the tool boundary. Invalid → tool returns error → model re-emits. No prose-discipline. This matches the ACI principle in `mission-control/CLAUDE.md` §"ACI design" — tool definitions are prompts that get more engineering than handler code.

**Concrete tasks**:

1. Write Zod schema in `src/tools/types.ts` or a new `src/reporting/contract.ts`. Decide which after spec §6 review.
2. Identify which existing tools emit aggregate-claim reports (likely candidates: anything that calls `infer()` to summarize stats). Survey `src/tools/builtin/` for tools matching the pattern; grep for `mc-ctl audit-claim` invocations.
3. Wrap those tools with the schema check. Failure mode: tool returns structured error pointing model at the missing field.
4. Test the boundary — fake report missing `verified_against:` → confirm schema error.
5. qa-auditor pass on the Phase 1 bundle.

**Done when**: schema rejects any report tool output that lacks `verified_against:`. CI test fails on absent-field reports.
