## Secondary check — k=2 stability rule

```sql
SELECT COUNT(*) FROM events
WHERE data LIKE '%replan_deferred%'
  AND timestamp > datetime('now','-7 days');
```

Result: **0 events**.

Interpretation: the k=2 rule (soft votes require 2 consecutive before replan) has NOT been exercised in this window. This is NOT a bug — soft votes fire only when `tool failure rate` or `tool-calls-per-goal` cross a heuristic threshold. All 8 runs in the window succeeded with 100% goal completion, so no soft votes were triggered; no soft votes means no opportunities to defer.

Consistent with the primary finding: Prometheus is currently running on tasks it handles cleanly, no replanning drama.

## Sample-size caveat

n=8 is small. Under the paper's null hypothesis (no lift) we'd expect the observed distribution. Under an alternative hypothesis (5-10% wide-gap rate) we'd need ~50-100 runs to distinguish confidently. At current Prometheus traffic (~8 runs/week), collecting that sample takes 6-12 weeks.

Decision principle: **close the thread NOW with the data we have rather than block indefinitely on a sample that may never materialize**. The paper (which used 155 tasks) already concluded Sonnet tournament lift is not significant (+4pp, p=0.21). Our 8-row sample is consistent with that null result. Extrapolating further doesn't change the expected value of shipping tournament mode: high 6× compute cost, no demonstrated quality gain.
