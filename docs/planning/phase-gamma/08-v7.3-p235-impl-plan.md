## Phase 5 — GEO Depth

### New tool 1: `seo_llms_txt_generate`

**Input**: sitemap URL or root domain. **Output**: `/llms.txt` file content per the emerging spec (https://llmstxt.org): H1 site name → blockquote summary → H2 section lists → title + URL + optional one-line description.

**Implementation**:

- Fetch sitemap (`robots.txt` → `Sitemap:` directive, fallback `/sitemap.xml`, `/sitemap_index.xml`).
- Extract `<loc>` entries (XML parse, recursive if sitemap index).
- Cluster URLs by first path segment (`/blog/*`, `/products/*`, etc.).
- For root domain: fetch `<title>` / meta-description for the summary.
- Emit markdown per spec. Return the string + a persisted filename suggestion.
- No new deps: use existing `webReadTool` / `validateOutboundUrl` / node's `DOMParser` equivalent (regex XML parse is fine for sitemaps).

### New tool 2: `seo_robots_audit`

**Input**: URL. **Output**: report of AI-bot coverage in `robots.txt` — which bots are allowed, which are blocked, which aren't mentioned at all. Distinguishes training-crawlers (e.g. `GPTBot`, `ClaudeBot`, `Google-Extended`) from user-triggered citation-crawlers (e.g. `OAI-SearchBot`, `PerplexityBot`).

**Implementation**:

- Fetch `{origin}/robots.txt` via `validateOutboundUrl` + `fetch`.
- Parse per RFC 9309: group by `User-agent:` lines, collect `Allow`/`Disallow` directives.
- Cross-reference against `ai-bots.ts` reference library (see below).
- Emit: `{allowed: [...], blocked: [...], unmentioned: [...], warnings: [...]}`.
- Warnings: "GPTBot blocked but Google-Extended unblocked" (contradicts intent), "ClaudeBot not mentioned" (default-allowed, probably unintended).

### New reference library: `src/tools/builtin/ai-bots.ts`

Static const array of 27 entries. Each: `{name, user_agent, operator, purpose: "training"|"citation"|"both", docs_url}`. Source: Auriti Labs reference.

Covers: OpenAI (GPTBot, OAI-SearchBot, ChatGPT-User), Anthropic (ClaudeBot, anthropic-ai), Perplexity (PerplexityBot), Google (Google-Extended, GoogleOther), Microsoft (Bingbot, MSNBot), Apple (Applebot), Meta (FacebookBot, Meta-ExternalAgent), and ~15 others.

### Extension: `seo_page_audit` — Princeton content-quality signals

Add a new `content_quality` section to the audit report alongside the existing structural score. Signals from Princeton KDD 2024 (https://arxiv.org/abs/2409.09978):

| Signal            | How to compute                                                                              |
| ----------------- | ------------------------------------------------------------------------------------------- | ------- | --- | ---------------- |
| Citation density  | count of `[N]` / `(Author, Year)` / hyperlinks / 1000 words                                 |
| Statistic density | regex `\d+(\.\d+)?\s\*(%                                                                    | percent | ×   | x)` / 1000 words |
| Quote presence    | `"..."` blocks or `>` blockquote — boolean + count                                          |
| Readability       | Flesch-Kincaid grade level (existing syllable counter OR port openai-readability heuristic) |
| Keyword stuffing  | top-noun repetition ratio — flag if any single noun >3% of body                             |

Impact-weighted scoring: add a `content_quality_score` (0-100) alongside `structural_score`; surface as separate field in return JSON. Don't merge into existing score to preserve back-compat of structural audit consumers.

### Scope regex

Extend `src/messaging/scope.ts` SEO group (current: `seo|rankings?|keywords?|serp|schema|structured data`):

Add: `llms\.txt|ai-?bots?|geo|readability|content quality|robots\.txt|citation density`.

### DB

No schema changes — `seo_audits` already has `findings` JSON column; put `content_quality` in there.

### Tests

~6 unit tests: sitemap extraction (flat + index), llms.txt format compliance, robots.txt parsing, AI-bot cross-reference, Princeton signal computation against a known fixture, scope regex assertions.

## Phase 2 — SEO Telemetry

### New tool: `seo_telemetry`

**Input**: `{url: string, include?: ("psi"|"gsc")[]}`. Default: both. **Output**: unified report.

**Two sub-engines:**

**1. PageSpeed Insights (PSI)**:

- Public API: `https://www.googleapis.com/pagespeedonline/v5/runPagespeed?url={URL}&strategy=mobile&key={KEY}` + desktop pass.
- API key optional (anon is rate-limited but works).
- Extract: Core Web Vitals (LCP, INP, CLS), Lighthouse scores (performance/accessibility/best-practices/seo), render-blocking resources.
- Env var: `GOOGLE_PAGESPEED_KEY` (optional).

**2. Google Search Console (GSC)**:

- Use the `gws` dispatch tool pattern: `gws searchconsole queries` via subprocess with OAuth token injection from `getAccessToken()`. Confirms hypothesis that gws Discovery covers GSC — if it doesn't (low probability, but check), write a minimal native adapter using googleapis discovery.
- Query: last 28 days, grouped by `query` + `page`, limit 100 by clicks desc.
- Extract: clicks, impressions, CTR, avg position per (query, page).
- Requires OAuth scope `https://www.googleapis.com/auth/webmasters.readonly` — verify scope already granted in `src/google/auth.ts`.

**Graceful degradation**: PSI fails (no network / rate limit) → return `{psi: {error}}`, don't block GSC. GSC fails (not OAuth'd) → return `{gsc: {error: "GSC not authorized…"}}`, don't block PSI.

### New table: `seo_telemetry_snapshots`

```sql
CREATE TABLE IF NOT EXISTS seo_telemetry_snapshots (
  id                 INTEGER PRIMARY KEY,
  url                TEXT NOT NULL,
  captured_at        TEXT DEFAULT (datetime('now')),
  psi_lcp_ms         INTEGER,
  psi_inp_ms         INTEGER,
  psi_cls            REAL,
  psi_perf_score     INTEGER,
  psi_seo_score      INTEGER,
  gsc_clicks_28d     INTEGER,
  gsc_impressions_28d INTEGER,
  gsc_ctr_28d        REAL,
  gsc_top_queries    TEXT,  -- JSON array
  raw                TEXT   -- JSON blob of full response
);
CREATE INDEX IF NOT EXISTS idx_seo_telemetry_url ON seo_telemetry_snapshots(url, captured_at DESC);
```

### Scope

Extend SEO group: `pagespeed|psi|core web vitals|lcp|cls|inp|lighthouse|search console|gsc|impressions|clicks|ctr|posición` (EN+ES).

### Tests

~5 unit tests: PSI adapter parses Lighthouse response, GSC gws dispatch builds correct args, graceful-degrade on PSI fail, scope activation, snapshot persisted.

## Phase 3 — AI Overview Monitoring

### Fetching strategy (risk-flagged)

Google SERPs don't expose AI Overview via any public API. Options:

1. **Stealth-browser fetch Google SERP HTML** (existing `src/lib/stealth-browser.ts` + fingerprint-injector) — detect AI Overview via DOM markers (`[data-attrid="AIOverview"]` / `.hdzaWe` etc. — brittle, markers change).
2. **SerpAPI / Serper.dev** — paid, reliable. Requires API key + budget approval.
3. **Defer Phase 3 entirely** until we pick a fetcher.

**Plan**: ship option 1 as MVP. If stealth fetch fails or returns anti-bot challenge, log and return `{error: "SERP fetch blocked", suggestion: "set SERP_API_KEY for SerpAPI fallback"}`. Don't throw — the tool degrades, doesn't break. Add SerpAPI path as a follow-up when a key is provisioned.

### New tool: `ai_overview_track`

**Input**: `{query: string}`. **Output**: `{present: boolean, sources: [{title, url, rank}], serp_top_10: [{title, url}]}`.

### New table: `ai_overview_tracking`

```sql
CREATE TABLE IF NOT EXISTS ai_overview_tracking (
  id           INTEGER PRIMARY KEY,
  query        TEXT NOT NULL,
  captured_at  TEXT DEFAULT (datetime('now')),
  present      INTEGER NOT NULL CHECK(present IN (0,1)),
  sources      TEXT,  -- JSON: [{title, url, rank}]
  serp_top     TEXT,  -- JSON: top 10 organic results
  fetch_status TEXT   -- "ok" | "blocked" | "error:<code>"
);
CREATE INDEX IF NOT EXISTS idx_aio_query ON ai_overview_tracking(query, captured_at DESC);
```
