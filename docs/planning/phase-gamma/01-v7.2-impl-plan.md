## §0 Scope-shift decision (read this first)

Roadmap v7.2 lists 4 items:

1. Graphify MCP integration (plumbing + tools)
2. CRM entity graph (prospects, deals, conversations)
3. Codebase graph (mission-control source)
4. Cross-source queries

**Shipping MVP = item 1 only.** Items 2–4 deferred to **v7.2.1** with written triggers. Reasoning:

| Item                 | Blocker                                                                                                                                                                                     | Resolution path                                                                |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| Codebase graph       | Upstream issue #451: duplicate nodes from parallel semantic subagents at scale. Unvalidated on 360-file corpus. First build costs Anthropic tokens (semantic pass is the bottleneck).       | v7.2.1 after #451 fix confirms, OR after local trial on 30-file subset scales. |
| CRM entity graph     | graphify has no first-class JSON extractor. CRM data lives in crm-azteca SQLite. Either build a JSON extractor upstream (out-of-repo) or write a md-export pipeline (crm → markdown dumps). | v7.2.1 + crm-azteca maintenance window. Requires md-export design.             |
| Cross-source queries | One graph per MCP process (confirmed from `serve.py`). Multi-corpus unified query = separate tool that routes to multiple servers + merges results. Non-trivial UX surface.                 | v7.2.1 after codebase + CRM graphs both exist.                                 |

**MVP corpus**: `/root/claude/mission-control/src/` (~360 TS files, excluding `*.test.ts` via `.graphifyignore`). **AST-only** — no LLM semantic pass.

**Recon pivot (during step 4)**: original plan was docs-only markdown corpus, but graphify's `collect_files` only handles code extensions (`.py .js .ts .tsx .go .rs .java .c .h .cpp .cc .cxx .hpp .rb .cs .kt .kts .scala .php .swift .lua .toc .zig .ps1 .m .mm`). Markdown content is extracted only via the skill-driven LLM semantic pass — too expensive for MVP. Code-only corpus is the cheapest path that produces a non-empty graph.

Why this scope change is safe: issue #451 (duplicate nodes) is **LLM-pass-specific** — it describes parallel semantic subagents colliding. AST extract is deterministic single-process. Running AST-only on 360 files has no #451 exposure. Trial on `src/finance` (72 files) during recon produced 411 nodes / 795 edges cleanly in under 5s.

## §2 Reference architecture (from recon)

Graphify is **two phases**:

- **Build** (skill-driven, Anthropic-cost): AST extract → LLM semantic extract → Leiden cluster → export `graph.json` + HTML/Obsidian. Requires ANTHROPIC_API_KEY. Runs once per corpus + incrementals via `--update` with SHA256 cache.
- **Serve** (pure NetworkX, zero LLM): `python3 -m graphify.serve <graph.json>` loads a prebuilt graph into memory and exposes 7 MCP stdio tools. Read-only query path.

`serve.py` enforces `validate_graph_path()` which requires the graph path to resolve inside a `graphify-out/` directory relative to CWD. Mitigated by setting `cwd` in the MCP server config to the corpus root (`./data/graphify/docs/`).

## §5 Risks & mitigations

| Risk                                                                                          | Mitigation                                                                                                                     |
| --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------- |
| `graphify build_graph()` Python API doesn't exist as a simple library call (skill-only build) | Step 4 inspects actual module. Fallback: drive build directly from this Claude Code session using graphify's internal modules. |
| `validate_graph_path()` rejects our layout even with `cwd` set                                | Test standalone (step 6) BEFORE wiring MCP. If rejected, patch with symlink or upstream monkey-patch.                          |
| Build phase costs more Anthropic tokens than expected                                         | Docs-only corpus (~30 small md files). Hard scope cap. If first run exceeds expectations, abort + reduce corpus size.          |
| Issue #451 (dup nodes) corrupts the docs graph                                                | Docs corpus is small enough to avoid parallel-subagent fan-out. Monitor graph_stats output for duplicate labels.               |
| Scope regex false-positives (e.g., "graph" in "bar graph", "graphic design")                  | Negative lookahead `graph(?!ic                                                                                                 | eme)`; add explicit negative tests. |
| MCP lazy mode doesn't fire on first tool call                                                 | Existing playwright lazy pattern is the reference; test matches that flow.                                                     |
| Graphify upstream ships a breaking change between v0.4.23 and next build                      | Pin version in install command: `graphifyy==0.4.23`. Bump deliberately in future sprints.                                      |
| Python venv makes deploy machine-specific (doesn't port to other environments)                | Acceptable — this is a single-VPS deployment. `.gitignore venv/` keeps the tree clean; document bootstrap step in README.      |
