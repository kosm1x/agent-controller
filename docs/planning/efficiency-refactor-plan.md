# Efficiency & Simplicity Refactor Plan

**Date:** 2026-07-05 · **Basis:** 5-agent adversarial audit (hot path / tools / data access / support subsystems / domain modules) over 142,528 source LOC, 530 files. All findings below carry auditor-verified file:line evidence; ~15 dead-code false positives were eliminated by re-checking `scripts/` CLI entry points and dynamic `import()` wiring before anything made this list.

## Executive verdict

The codebase is **fundamentally sound where it matters most** — the per-message hot path is correctly parallelized and cache-disciplined (82% cache-read), DB indexing is strong (exactly one gap), API auth is clean, and the 39k-LOC tools layer is breadth, not bloat. A senior engineer would sign off on the architecture and then flag, in order:

1. **One live High bug**: video assembly blocks the entire orchestrator event loop for 2–4 min per video (`execFileSync` ×5).
2. **~1,855 LOC of confirmed-dead modules** (finance optimizer triangle ~940; adr-writer 285; skill-eval-loop 275; memory/consolidate.ts 197; prometheus/resume.ts 158) plus a dead table with a lying doc comment.
3. **Zero shared tool infrastructure**: 3 coexisting error conventions (parsed all-three by task-executor to detect mutation failure), 13 hand-rolled fetch skeletons, 153 hand-rolled `err instanceof Error` extractions, 189 tool names hand-synced in two places with no guarding test.
4. **Per-message allocation waste**: ~150 Zod schema objects rebuilt per message for static tools; a ~150-name Set rebuilt per message for a log line; 5 regexes compiled per message; 2 SHA-256 hashes of a 34KB prompt per message for a concluded diagnostic.
5. **Structural debt that grows**: no DB migration versioning (34 boot-time ALTER probes, forever); no retention (tasks+runs = 278 MB of a 451 MB DB, unbounded); a 905-line `handleInbound` god-method; ~600 inert-in-production LOC dominating `adapter.ts`.

**Honest efficiency assessment**: runtime perf is already good. The video sync-exec is the only defect a user would _feel_. Most wins here are CPU-per-message trims, unbounded-growth caps, and maintainability — the "most efficient code use" goal is served primarily by deleting ~7–9k LOC (dead + duplicated), not by micro-optimizing.

**Load-bearing constraint for ALL phases**: never vary the stable system-prompt prefix (`CACHE_BREAK_MARKER` split, `cacheable:false` routing at fast-runner.ts:851/863/876). Every change must be prompt-neutral or stay on the variable side. Post-deploy, watch cache-read % via `mc-ctl briefing-gate` cohort.

**What was audited and cleared** (no action): api/ auth (all routes gated, constant-time compare), boot path (lazy cron imports), skills/ loader (boot-once, not per-call), backtest cluster (live — earlier "dead" read was a `grep -v test` artifact), `intel/` vs `intelligence/` (different domains, not overlap), MemoryService/XBackend abstractions (2 real impls each), dormant-by-design gates (V83, self-healing, v82 — gating is correct).
