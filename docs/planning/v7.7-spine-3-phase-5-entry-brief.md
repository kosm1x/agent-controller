## What this phase is

**S5 Phase 5 — Operator surface + activation gate** — adds the operator-facing tooling to certify, archive, and observe skills, plus authors ≥5 production-grade skill bodies that pass the activation gate.

**Spec sections to read first**: `docs/planning/v8-substrate-s5-spec.md` §7 (lifecycle — `skill_archive`, `skill_revert`), §13.5 (Phase 5 task list), §14 (activation gate query + watchpoints), §15 Q5/Q6/Q7. Reading order: §13.5 → §14 → §7 → §15.

- `mc-ctl skills revert <name> <version>` — point current_version_id at an older version

- Optional: frontend `/skills` page for browsing certification status, recent runs, test pass rate
- **5 production-grade skill bodies authored + certified** (the actual gating work)

## Anti-mission reminder

> Phase 5 ALLOWS authoring of 5 production skills — this is the ONE phase of Spine 3 where the anti-mission relaxes. The 5 skills are concrete deliverables required by the V8.0 activation gate.

If a bundle introduces MORE than the 5 targeted skills, push back. Operator picks the 5 high-leverage candidates from the existing 57 legacy skills OR ritual workflows; you author the frontmatter + body + tests + run the cron sweep to certify.

The activation gate query (spec §14):

```sql
SELECT COUNT(*) FROM skills WHERE is_certified=1 AND active=1
  AND EXISTS (SELECT 1 FROM skill_test_runs
              WHERE skill_id = skills.skill_id
              AND result='pass' AND ran_at >= datetime('now','-7 days'))
```

Currently returns **0**. Phase 5 close = this query returns ≥5.

## Risks specific to this phase

| Risk                                                                                         | Severity | Mitigation                                                                                                                                                    |
| -------------------------------------------------------------------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Critic rejects all 5 candidate skills → activation gate fails                                | High     | Pick candidates from existing 57 (already battle-tested). Operator override available for any borderline rejection. Document override reasoning in fmtter.    |
| Test sweep doesn't run within 7 days of certification → recent_passes < test_count           | Medium   | Cron sweep is 6h cadence (Phase 2 B2). 7-day window admits 28 sweep cycles per skill. Won't be tight.                                                         |
| 5 authored skills' inputs declarations conflict with how legacy callers invoke them          | Medium   | Phase 5 authors NEW frontmatter from scratch — doesn't migrate the legacy skill calling pattern. Old `findSkillsByKeywords` retrieval path stays as fallback. |
| Skill body adversarially injects `{error: "HARNESS_OVERRIDE_ATTEMPT"}` to trigger anti-list  | Low      | RUNNER_PREFIX explicitly detects this and returns a structured error envelope. Body-side adversarial behavior surfaces clearly in skill_test_runs.            |
| `mc-ctl skills certify <name>` operator surface conflicts with cron-sweep auto-certification | Medium   | Document both paths: manual `certify` is an OVERRIDE that fires immediately; sweep certification is the autonomous path. They don't conflict; they layer.     |
