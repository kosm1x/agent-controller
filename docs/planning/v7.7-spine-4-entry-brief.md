## What this spine is

**Conway Pattern 1 — General-events middle layer.** Today episodic memories live as raw events (journal, cost_ledger, recall_audit, decision_events); semantic memories live as facts (jarvis_files). There is NO middle layer that aggregates events into "what's been happening this week" form. V8.1's morning-brief proactive scan needs that layer.

**Reference**: `~/.claude/projects/-root-claude/memory/reference_conway_2005_sms.md` — read this first; it's the empirical grounding for why the middle layer matters.

**Why it's substrate, not capability**: V8.1's spec explicitly cites Pattern 1 as a structural prerequisite. Spine 4 builds the QUERYABLE LAYER; it does NOT build the morning-brief that consumes it (that's V8.1 capability work — anti-mission).

## Shape (from V7.7-GUIDE §"Spine 4")

Derive a `general_events` view (or table) over: `journal` + `cost_ledger` + `recall_audit` + `decision_events`. Time-windowed aggregations: per-day, per-week, per-project. These become the inputs to V8.1's morning-brief proactive scan.

**Done when**: V8.1's Phase 1 spec can run against this layer without further build-out.

## First concrete steps after `/compact`

1. **Read** `reference_conway_2005_sms.md` (the Conway 2005 SMS paper notes) — ~10 min cold.
2. **Grep** the 4 source tables' schemas in `src/db/index.ts`: `journal`, `cost_ledger`, `recall_audit`, `decision_events`. Confirm they all exist and inventory their columns + timestamps.
3. **Decide view-vs-table**: a SQL view is zero-maintenance but recomputes on every query; a materialized table needs a refresh cron. Spine 2 (S3 drift detector) and Spine 1 set precedent for additive tables + cron sweeps. Pick based on query cost — if the per-week aggregation over journal+cost_ledger is cheap, a view; if not, a table refreshed by a cron (mirror `src/skills/test-sweep.ts`'s node-cron pattern).
4. **Check V8.1 spec** (`docs/planning/v8-capability-1-spec.md` if it exists) for the exact query shapes the morning-brief needs — design the layer to those, not speculatively.
5. **Plan Bundle 1** as a concrete task list; start ship-it once aligned.

## Anti-mission reminder

- ❌ Don't build the morning-brief itself. Spine 4 ships the events layer ONLY.
- ❌ Don't add capability behavior (no proactive scanning, no LLM summarization of events). The layer is a queryable aggregation, full stop.
- ❌ Don't over-design the time windows. Per-day / per-week / per-project is the spec; don't add per-hour / per-quarter speculatively.

## Operating rules that especially apply

1. **Audit every bundle** — qa-auditor on every ship (per V7.7-GUIDE).
2. **Schema discipline** — if a `general_events` TABLE is chosen, it's additive (`CREATE TABLE IF NOT EXISTS`); applies live, no DB reset. A VIEW is also additive. Per CLAUDE.md, never reset the DB.
3. **Bundle-slice cadence** — Spine 3 held ~2-2.5h per bundle across 10 bundles. Keep bundles small.

## Risks specific to this spine

| Risk                                                                | Mitigation                                                                                                                                     |
| ------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| The 4 source tables have inconsistent timestamp columns / timezones | Inventory each table's timestamp column first (step 2). mc runs `TZ=America/Mexico_City`; normalize windows to that.                           |
| A view over 4 joined tables is slow on large journal                | Measure before committing to a view; fall back to a refreshed table + index if the per-week query exceeds ~100ms.                              |
| `decision_events` table may not exist yet (V8.x table)              | Confirm in step 2. If absent, scope the layer to the 3 tables that exist + leave a documented seam for decision_events.                        |
| Over-coupling to V8.1's exact needs before V8.1 spec is firm        | Design to the V8.1 Phase 1 spec's query shapes (step 4); if that spec is not yet written, ship the obvious aggregations and document the seam. |
