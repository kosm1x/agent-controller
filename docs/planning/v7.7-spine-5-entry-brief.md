## What this spine is

**Conway Pattern 2 — Self-defining memory cohort + operator-profile skeleton (Q3 carve-out).**

Conway's empirical claim (`reference_conway_2005_sms.md` §Pattern 2): a small set of "self-defining memories" stays highly accessible because it grounds the system's self-image and long-term goals. V8.2's strategic proposals **fail without this** — without identity-grounding, proposals read as generic LLM output, not "Jarvis-with-history-of-this-operator." Pattern 2 is a hard V8.2 prerequisite.

Spine 5 builds the **stable cohort definition**: which projects, which contacts, which conversation threads are the operator's identifiable cohort — so V8.1's briefings (Phase 2-3) and V8.2's proposals can speak in those terms instead of generic "you have 3 stalled tasks" language.

**Plus the Q3 carve-out** — an `operator_profile` **skeleton** table. Schema only. No inference, no LLM-derived fields, no personality/preference modelling. V8.2 owns the semantics; v7.7's contract is just "the bone structure exists and V8.2 can extend it without a migration."

## Shape (from V7.7-GUIDE §"Spine 5")

1. **Self-defining cohort** — derive cohort membership from existing data: NorthStar projects, frequent CRM contacts, recurring conversation threads. Daily roll-up. "No new schema if existing data supports it; one small denormalized table if not."
2. **`operator_profile` skeleton** — `(cohort_member_id, attribute_key, attribute_value, written_at, written_by)`. Write-side populated **ONLY** by the Conway Pattern 2 cohort roll-up. NO inference. NO LLM-derived fields.

**Anti-mission compliance test for the skeleton**: can V8.2 redefine the value-side semantics (`attribute_key` / `attribute_value` meaning) without migrating data? If yes — the skeleton is thin enough. If no — revert and re-scope.

**Done when**: V8.1's Phase 2-3 (where cohort framing kicks in) can compose against the cohort definition without further design. AND the `operator_profile` skeleton table is shipped with a write-path test proving it is never populated except by the cohort roll-up.

## Anti-mission reminder

- ⚠️ **operator-profile is SKELETON ONLY** — storage shape + read API. NO inference, NO personality/preference modelling, NO write-side population beyond the cohort roll-up. If a field's value would be LLM-derived, it does not belong in v7.7.
- ❌ Don't build V8.2's proposal layer. Spine 5 ships the cohort + the empty profile scaffold.
- ❌ Don't over-design the cohort. NorthStar projects + frequent contacts + recurring threads — daily roll-up. No richer signal modelling.

## Operating rules that especially apply

1. **Audit every bundle** — qa-auditor R1 per ship; R2 only on Critical-bearing/contract-drift bundles (the Spine 3/4 cadence held: 0 R2 needed across Spine 4's 3 bundles).
2. **Schema discipline** — additive only (`CREATE TABLE IF NOT EXISTS`); never an enum-CHECK alteration (that needs a destructive recreate — defer to v8.0). Never reset the DB.
3. **Spec-vs-codebase reconciliation** — Spine 4's lesson: the spec (Conway reference / V8.2 spec) may name tables/columns that do not exist or that the codebase models differently. Design to ground truth; document every deviation in the impl-delta doc.
4. **Bundle-slice cadence** — keep bundles small (~2-2.5h, ~300-500 LOC each).
