## §1 — What shipped

Conway Pattern 1: the **general-events middle layer** — a queryable tier
between abstract knowledge (NorthStar, MEMORY.md) and raw episodic chunks
(tasks, conversations). V8.1's morning-brief proactive scan retrieves at the
general-event level and descends to episodic only as needed, instead of
drowning in a 50-item episodic recall.

## §2 — Substrate live (post-Spine-4)

- **Schema** — `general_events` (the ~30-50 row middle-layer cohort) +
  `general_event_episodic_links` (descent edges). Additive migration in
  `src/db/index.ts`; applies live on next boot.
- **Write path** (`src/events/general-events.ts`) — `createGeneralEvent`,
  `getGeneralEvent`, `listGeneralEvents`, `updateGeneralEvent`,
  `archiveGeneralEvent`, `supersedeGeneralEvent`, `linkEpisodic`,
  `unlinkEpisodic`, `getEpisodicLinks`. Summary embedded on write;
  `episodic_count` recomputed from the links table (cannot drift).
- **Read path** (`src/events/retrieval.ts`) — `retrieveGeneralEvents`
  (Layer 1: archived/objective/window filter, cosine-or-recency ranking),
  `retrieveForBriefing` (Layer 2: descend to episodic samples — the V8.1
  briefing entry point), `resolveEpisodic` (resolve a link to its source
  row, all 6 kinds, never throws), `aggregateGeneralEvents` (pure per-day /
  per-week / per-project roll-up).
- **Operator surface** — `mc-ctl events {list,show,retrieve,aggregate,seed}`.
- **Observability** — `mc_general_events_ops_total` counter
  (created|archived|retrieved).
- **Seed** — `scripts/general-events-seed.ts` populates the v7.x
  substrate-arc cohort (~12 events).

## §3 — Deviations from spec §5 (intentional, documented)

| Spec §5 says                                                                            | Spine 4 shipped                                     | Why                                                                                                                                                                                                                         |
| --------------------------------------------------------------------------------------- | --------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CREATE VIRTUAL TABLE ... USING vec0`                                                   | `summary_embedding BLOB` column + in-memory cosine  | The codebase uses BLOB + cosine everywhere (`conversation_embeddings`, `skills.description_embedding`); `sqlite-vec` is "a future option when scale > 1000". `general_events` is a ~30-50 row cohort. Build-once-use-twice. |
| Two embeddings (`summary` + `details`)                                                  | `summary_embedding` only                            | Descent runs through the explicit `general_event_episodic_links` table, not a second vector probe. `details_embedding` is re-addable as an additive `ALTER TABLE`.                                                          |
| Embedding dim `FLOAT[1024]`                                                             | 1536-dim (Gemini `gemini-embedding-001`)            | Matches the actual embed model the codebase uses. The spec's 1024 was illustrative.                                                                                                                                         |
| Population: manual seed **+** auto-discovery (nightly LLM cluster → S2 critic → insert) | Manual seed only                                    | The Spine 4 anti-mission bars LLM event-summarization. Superset, no conflict.                                                                                     |

## §4 — "Done when" — met

The GUIDE's Spine 4 done-when: _"V8.1's Phase 1 spec can run against this
layer without further build-out."_ V8.1 spec §5's retrieval contract is
`retrieveForBriefing(context, k)` → `{ generalEvents, episodicSamples }`.
That function exists, is tested, and composes the two-layer retrieval the
spec describes. The `episodic_kind` enum matches spec §5 exactly. The
manual-seed cohort gives V8.1 non-empty retrieval to compose against. No
further substrate build-out is needed for V8.1 Phase 1 — only the
capability work (the briefing judgment itself) remains, which is V8.1's job.

The GUIDE's loose "view over journal + cost_ledger + recall_audit +
decision_events" phrasing was superseded by spec §5: `journal` and
`decision_events` are not tables in this codebase, and §5's
`episodic_kind` enum (`task|conversation|memory_item|recall_audit|
cost_ledger|report`) names only sources that exist. Designed to §5.

- **`mc-ctl events aggregate --by=week`** — 4 week-buckets, Monday-keyed, counts correct.
- **`mc-ctl events show v7-7-spine-4-general-events`** — full detail + summary + (empty) links render correctly.

**One bug found + fixed during deploy** — `mc-ctl events seed` initially aborted: mc-ctl runs `set -euo pipefail`, and the Bundle-3 R1-W3 fold sourced `.env` via `. ./.env` — a `.env` value containing a `$N` token is an unbound-variable fatal under `set -u`. Replaced the source-`.env` approach with an `env_value()` helper that greps a single key (`GEMINI_API_KEY`) out of `.env` without sourcing it, mirroring mc-ctl's existing `api_key()`. Shipped in the closure commit. Lesson: never `source` a `.env` from a `set -u` script — extract keys with `grep`.
