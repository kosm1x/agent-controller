## 7b. Phase 6 — Task-trace observability (merged 2026-07-13 from plan-9-10 Sprint 1)

The one genuinely new workstream in Jarvis's self-development plan (`jarvis-kb/projects/agent-controller/plan-9-10.md`), and it names a real gap V8.5 didn't cover.

**Reality check first** (his "observabilidad 3/10 — caja negra total" overstates): mc-prometheus is live with an exporter, KB-drift/cost/recall alert rules, cost_ledger writes per-run rows (including Hindsight via the 5-min pull), and scope_telemetry + recall_audit exist. What does NOT exist is **per-task trace correlation**: no structured `TaskStarted → RoundCompleted → ToolCalled → TaskDone/Failed` event stream keyed by task_id, no replay, no "which tool/round killed this task" answer without reading raw journalctl. That gap is real — every incident forensics session this month has been journalctl archaeology.

**Deliverable**: structured trace events correlated by task_id (round, tool, tokens in/out, latency, accumulated cost per step); `mc-ctl trace <task_id>` renders the full replay; a failed task's trace shows the terminal round/tool. §17 verdict-rate-over-time (plan-9-10's "dashboard" ask) becomes a query over the same store, not a new UI.

**Stack decision — deliberately NOT Langfuse-first.** plan-9-10 proposes OpenTelemetry + self-hosted Langfuse (new Docker stack + UI under Caddy). Default here is the simplest viable version: **a SQLite trace table + the `mc-ctl trace` viewer — zero new services** on a RAM-constrained VPS that saturated yesterday. Design the emit seam OTel-shaped (one `emitTraceEvent()` at the runner/dispatcher seam — instrumentation at the stable seam, per the instrumentation-backend-coupling lesson) so a Langfuse/OTel exporter can be bolted on later WITHOUT re-instrumenting, and escalate only if the SQLite viewer proves insufficient in practice.

**Sequencing**: after Phase 3 (3.3 cost-ledger completeness feeds the per-step cost trace); independent of Phase 5 but strengthens it (trace evidence backs §14 shadow-decision review).
