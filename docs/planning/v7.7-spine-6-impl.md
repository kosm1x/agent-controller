## §2 — Substrate live (post-Spine-6)

- **Named recall modes** (`src/memory/recall-mode.ts`) — `coherence` | `correspondence` | `unfiltered`. `RecallOptions.recallMode` is the additive high-level option; the legacy `excludeOutcomes`/`includeFailed` knobs still work and take precedence. `resolveExcludeOutcomes` (the exclude set) and `resolveRecallMode` (the audit tag) share precedence so the logged mode can never contradict the filter applied.
- **`recall_audit.mode` column** — every recall through `HindsightMemoryBackend` is tagged with its resolved mode.
- **`recall_coherence_suppression_rate` drift signal** — the 14th seed signal in Spine 2's registry; `weekly` cadence; the correspondence audit composed onto the S3 substrate (Spine 2's scheduler+evaluator runs it — no new cron).
- **`mc-ctl recall-modes`** — read-only view of the recall-mode distribution + the suppression rate.

## §3 — Design decisions

| Decision                                                                                    | Rationale                                                                                                                                                                                                                             |
| ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Additive `recallMode`, NOT a rename of `excludeOutcomes`**                                | The entry brief floated renaming `excludeOutcomes` → `recall_mode`; additive is the safer call it also flagged — zero callers break, the legacy knobs stay as low-level overrides, `coherence` stays the default → behaviour-neutral. |
| **The "weekly correspondence-audit cron" is a weekly-cadence drift signal**, not a new cron | Genuine composition with Spine 2: a `cadence='weekly'` drift signal IS evaluated weekly by Spine 2's scheduler. The GUIDE's "composes with Spine 2" is satisfied without rebuilding alerting.                                         |
| **The signal seeds `enabled=0` (disabled-pending)**                                         | See §4 — `recall_audit` is dormant under the Hindsight demote. Mirrors Spine 2's pattern for its 10/13 substrate-less signals.                                                                                                        |
| Suppression rate = `Σexcluded / Σ(excluded+kept)`                                           | Deterministic, SQL-expressible, no LLM. A rising rate = coherence suppressing more = Conway's drift.                                                                                                                                  |

## §4 — The recall_audit dormancy finding

`recall_audit` has **1460 rows but none since 2026-05-10**. Root cause (pre-existing, NOT introduced by Spine 6): the Hindsight demote (queue #15) routes recall through `SqliteMemoryBackend`, which does not call `logRecall`/`applyOutcomeBias` — only `HindsightMemoryBackend` does, and `HINDSIGHT_ENABLED=false`. So `recall_audit` logging **and** the coherence outcome filter are both dormant in the current production config.

Spine 6's substrate is correct regardless — it activates the moment recall logging resumes. The `recall_coherence_suppression_rate` signal is seeded `enabled=0` with the activation trigger documented in `notes`; the baseline `0.0406` is real (measured from the 1460 historical rows). Restoring recall logging — wiring `logRecall` into `SqliteMemoryBackend` — was deliberately NOT done: it changes live recall behaviour and touches the Hindsight-demote routing, both outside Spine 6's substrate scope. Queued as **`S6-recall-audit-dormant`**.
