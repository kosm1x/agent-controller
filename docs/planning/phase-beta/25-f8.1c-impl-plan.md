# F8.1c — PM-Native Cadence + NO-Side Shorting + Quote Refresh Impl Plan

Session 89 · 2026-04-21 · β-addendum closer (post-F9, Phase γ active).

## §0 Scope & rationale

**Operator constraint (2026-04-21)**: "Weekly is good enough for equities and macro variables, but not for PolyMarket."

F8.1a (PM alpha layer, 2026-04-20) and F8.1b (PolymarketPaperAdapter, 2026-04-20) shipped the PM pipeline on a weekly cadence inherited from the F7/F7.5 weekly-equity lock. That was the right default for v1 — PM events resolve irregularly and paper trading doesn't need to chase every tick. But prediction markets _do_ move hourly on news and resolve on their own clocks, so weekly rebalance leaves signal on the table and exposes stale-position traps when a market resolves mid-week.

This sprint closes the cadence gap plus the two load-bearing v1 deferrals: **NO-side shorting** (already-computed negative α-weights currently clipped to zero) and **automatic quote refresh** (stale quotes would abort every daily rebalance otherwise). Firewall/orderbook/MCP/cross-venue items from the F8.1 deferral list stay deferred with written triggers — they don't block daily PM operation, and bundling them would stretch this from one session to four.

## §2 Scheduling mechanism

Per recon: there is **no `scheduled_tasks` table**. Rituals are wired via `node-cron` expressions listed in `src/rituals/config.ts`, consumed by `src/rituals/scheduler.ts`. Adding a daily PM cadence = one new entry in `config.ts`. No schema change, no new scheduler infra.

**Cron**: `0 6 * * *` in `America/New_York` (= 05:00 CDMX under standard time, 04:00 under DST for MX). Runs daily including weekends (PM markets don't follow equity calendar). Timezone set per-ritual — `isNyseTradingDay` gate explicitly NOT applied (v7.0 F9 feedback memory: per-ritual tz is a known F9 pattern; PM runs even on weekends).

## §3 NO-side shorting — detailed mechanics

### Today (v1)

```ts
const effectiveWeight = Math.max(0, weight); // negative → 0, pure exit
const symbol = `${marketId}:${primaryOutcome}`; // e.g., "0xM1:Yes"
```

Negative alpha weights (market priced above alpha's `p_estimate`) silently become exits. The Kelly math is already emitting `weight < 0` rows; the paper adapter just ignores them.

### After this sprint

```ts
// Determine side from weight sign
const side = weight < 0 ? "no" : "yes";
const notional = Math.abs(weight) * equity * (1 - CASH_BUFFER_BPS / 10000);
const outcome =
  side === "yes"
    ? primaryOutcome // usually "Yes"
    : findComplementOutcome(market, primaryOutcome); // usually "No"
const symbol = `${marketId}:${outcome}`;
```

Because PM markets are binary (YES + NO prices sum to ~1 before fees), "short YES" ≈ "long NO". For this v1 we express the short by **buying the NO token**, not by a synthetic short-sell. This sidesteps the need for margin accounting — our existing cash-based `placeOrder("buy", …)` path handles it.

**Edge cases**:

- Non-binary markets (3+ outcomes): skip the short. Log with exclusion reason `multi_outcome_short_unsupported`. Alpha tool already clips to binary markets at §0 of pm-alpha.ts so this is defensive.
- Already holding the YES token with positive balance when a new negative weight arrives: sell the YES (existing exit path) then buy NO (new open). Two-step, recorded as two fills.
- Already holding the NO token: extend / reduce as normal (treat as a `kind: polymarket` position of a non-primary outcome — discriminated-union guard already handles this).

## §4 Staleness thresholds

`PM_PAPER_QUOTE_STALE_MS` becomes a constructor param `quoteStaleMs?: number` on `PolymarketPaperAdapter`. Default 5d (preserves weekly behavior). Daily cadence ritual constructs the adapter with `quoteStaleMs = 24 * 60 * 60 * 1000`.

The abort-on-stale gate in `runPmRebalance` reads `staleMarkets` already populated from `getPositions` — no change needed there; the adapter-level threshold just gets tighter.

**Rationale for 24h not tighter**: Polymarket midpoints can go hours between trades on low-volume markets. 24h threshold catches "market resolved and we don't know" or "Polygon outage" scenarios without tripping on normal off-hours quiet.

## §5 Quote refresh pattern

The daily ritual runs three tools in sequence (single submission, LLM tool-loop, not separate schedules):

1. `prediction_markets limit=100` — fetches the top 100 active Polymarket markets fresh from the Gamma API on every call (the tool does not have a separate refresh action; "refresh" is a side-effect of invocation). Updates `prediction_markets.outcome_tokens` and `fetched_at`.
2. `pm_alpha_run` — reads the freshly-cached markets, computes weights (positive + negative), persists to `pm_signal_weights`.
3. `pm_paper_rebalance cadence=daily` — reads latest `pm_signal_weights`, rebalances paper portfolio, persists fills + thesis with `entry_signal='pm_daily_rebalance'`.

No new tool; reuses existing three. The ritual description teaches the LLM to run them in that order. The tool surface was verified against `src/tools/builtin/market.ts:1290` — parameters are `{query, category, event_id, limit}` with hard-cap 100.

## §8 Risks & mitigations

| Risk                                                                                     | Mitigation                                                                                                    |
| ---------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| Daily rebalance fires on weekends when PM activity is low → noise in thesis history      | Accept; PM is 24/7 and weekend-sensitive events (politics, sports) are exactly where weekly cadence was blind |
| NO-side fills distort portfolio accounting (YES+NO on same market → both positions long) | Both positions share `market_id`; aggregation is by `(account, market_id, outcome)` already; no double-count  |
| Quote refresh at step 1 fails → step 3 aborts on stale                                   | Ritual-level error surfaces to operator; next day's run retries; no positions mutated                         |
| 24h staleness too tight → daily rebalances abort on normal off-hours                     | Start at 24h; if abort rate > 10% in 14d, raise to 48h and document                                           |
| Operator invokes `pm_paper_rebalance` manually without `cadence` param                   | Default `"weekly"` preserves F8.1b behavior exactly                                                           |
